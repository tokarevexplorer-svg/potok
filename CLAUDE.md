# ПОТОК — Суперап

## Общее описание
Веб-приложение "Поток" — прод: https://potok-omega.vercel.app (бэкенд: https://potok-production-a2dc.up.railway.app), позже — собственный домен. Личный суперап с инструментами для жизни и работы. Владелец — Влад, радиожурналист, запускает блог в Инстаграме. Опыт разработки — нулевой, вся разработка ведётся через Claude Code.

## Навигация суперапа
Боковое меню auto-hide: скрыто по умолчанию, появляется при наведении мыши к левому краю экрана (на мобилке — бургер). Структура:
- **Блог**
  - **Инструменты** → **База референсов** (бывший "Аналитик", текущий проект)
  - **Команда** (будущее: AI-агенты — аналитик, редактор, креатор)
В будущем добавятся другие разделы суперапа: Финансы, Новости, Сохранёнки и другие.
Меню расширяемо — добавление нового раздела не требует переделки.

## ТЕКУЩИЙ ПРОЕКТ: Блог → Инструменты → База референсов

### Суть
Инструмент для анализа сохранённых Reels и постов из Инстаграма (ранее назывался "Аналитик"). Влад вставляет ссылку (или список ссылок) на видео или посты → система автоматически собирает данные, транскрибирует (для видео), анализирует через AI и складывает в таблицу с фильтрацией. Цель — разобрать тысячи сохранённых видео и постов, систематизировать и найти идеи для собственного блога. Поддерживает три типа контента: видео (Reels), фото-посты и карусели.

### Пользовательский сценарий
1. Открываю сайт → в левом меню нажимаю "Блог" → "Аналитик"
2. Нажимаю "Добавить видео" → вставляю ссылку на Reels → нажимаю "Анализировать"
3. Система обрабатывает (30–60 сек на одно видео): парсит данные, транскрибирует, прогоняет через AI
4. В таблице появляется новая строка со всеми данными
5. Я просматриваю, ставлю теги, пишу заметки, фильтрую, удаляю мусор
6. Массовое добавление: загружаю список ссылок (до 200–300 за батч) → фоновая обработка с прогресс-баром → можно закрыть браузер, результаты появляются по мере готовности

### Структура данных (столбцы таблицы)

#### Блок 1 — Справочная информация (заполняется автоматически)
- Дата публикации видео
- Ссылка на видео
- Автор (название блога)
- Ссылка на профиль автора
- Описание под видео (caption)
- Превью/обложка (миниатюра)
- Хронометраж (длительность в секундах, отображается как "0:45" или "1:23"). Для фото/каруселей — не применимо
- Тип контента: video / image / carousel (определяется Apify при парсинге). В таблице отображается иконкой или текстом вместо хронометража для фото/каруселей

#### Блок 2 — Статистика (заполняется автоматически)
- Просмотры
- Лайки
- Комментарии
- ~~Шеры (поделились)~~ — Instagram не отдаёт публично с 2023, столбец убран из UI (см. «Известные ограничения»)
- Коэффициент вирусности (расчётный, формула на основе просмотров/лайков/комментов относительно средних показателей аккаунта)

#### Блок 3 — Содержание (заполняется автоматически)
- AI-саммари: краткое описание о чём видео (2–3 предложения)
- Транскрипция: полный текст речи из видео. Если речи нет — значение "Без речи (музыка/визуал)"
- Категория AI: автоматическая категоризация. Стартовый набор категорий (AI может предложить новую если ничего не подходит):
  - Вайб-кодинг
  - История
  - Культура
  - Санкт-Петербург
  - Юмор
  - Лайфстайл
  - Бизнес / предпринимательство
  - Путешествия
  - Еда
  - Мотивация / саморазвитие
  - Технологии
  - Образование
  - Другое (с указанием предложенной категории)
- Референс для блога (`is_reference`, boolean): AI определяет, относится ли контент к блогу (приёмы, темы, авторы, визуал, контент-мейкинг) или это что-то личное (рестораны, выставки, мемы, бытовое). Для фото/каруселей — автоматически false. Отображается в таблице: ✅ референс / ❌ другое

#### Блок 4 — Ручные поля (заполняются пользователем)
- Категория Я: пользователь подтверждает категорию AI или ставит свою
- Теги (множественный выбор, можно ставить несколько, Notion-стиль с цветами):
  - 🎯 Тема — хочу сделать похожее по теме
  - 🪝 Хук — цепляющее начало, разобрать приём
  - 🎨 Визуал — красиво снято/смонтировано, повторить
  - ⭐ Автор — интересный блогер, следить
  - 🗑 Мусор — пометить для удаления
  - (пользователь может создавать новые теги в процессе работы)
- Заметка: свободное текстовое поле, необязательное (поддерживает голосовой ввод через микрофон)
- Оценка (эмодзи рядом с превью):
  - ✅ Верифицировано — видео просмотрено и подтверждено как полезное
  - 🔥 Супер — выдающееся видео, вернуться первым
  - 🔄 Повторить — снять такое же видео слово в слово

### Таблица закладок (bookmarks)
Отдельная таблица в Supabase для видео, которые не относятся к блогу, но которые Влад не хочет терять (рестораны, выставки, AI-контент и т.д.). Структура — копия основных полей из videos (url, author, description, thumbnail, ai_summary, ai_category, user_note). Наполняется из Tinder-режима (свайп влево → "В закладки"). На сайте пока не отображается — доступна через админку Supabase. В будущем станет фундаментом для раздела "Сохранёнки".

### Функциональность таблицы
- Дата публикации = гиперссылка на видео в Instagram (открывается в новой вкладке)
- Превью: клик = увеличение в модалке (лупа)
- Фильтрация по: категории AI, категории Я, тегам, автору, наличию/отсутствию транскрипции, оценке, типу контента (видео/фото/карусель), референсу (для блога / другое)
- Сортировка по: дате, просмотрам, лайкам, коэффициенту вирусности, хронометражу
- Поиск по тексту (в транскрипции, саммари, описании, заметках)
- Удаление: единичное и массовое (чекбоксы + "Выбрать все" + "Удалить выбранные" с подтверждением)
- Массовое перемещение в закладки: чекбоксы + "В закладки" (рядом с "Удалить" в плавающей панели)
- Фильтр "Мусор" + удалить все отфильтрованные одним действием
- Режим просмотра (Tinder-swipe) — кнопка "Режим просмотра" запускает полноэкранный просмотр с учётом текущих фильтров
- Режим "Очистка" — кнопка запускает упрощённый Tinder-swipe только по видео с is_reference=false: влево = в закладки, вправо = пометить как референс

### Системный промпт для AI-анализа
При обработке видео AI получает: описание (caption), транскрипцию (если есть), статистику. На выходе:
- Саммари (2–3 предложения, о чём видео)
- Категория из предложенного списка (или новая с пояснением)
- is_reference (true/false) — определяется на основе контекста блога (см. ниже)
Промпт должен быть на русском. Если транскрипция пустая — AI работает только по описанию и обложке. Для фото/каруселей — транскрипция не применима, AI работает только по описанию.

#### Контекст блога для AI-классификации (is_reference)

Блог исследует историю и культуру (преимущественно России и русскоязычного пространства) через короткие журналистские сюжеты. Главный принцип — точки поворота: события, ошибки, запреты, странности и случайности, которые повлияли на культуру, общество или повседневность сильнее, чем кажется на первый взгляд. Подача — журналистский репортаж, серьёзная подача с ироничной сутью, в стиле Парфёнова. Циклы: исторические запреты в России (главный), исторические ошибки, культурные и городские странности (Петербург), возможный сатирический слой.

**is_reference = true (брать) если видео про:**
- Историю России, СССР, Российской империи: события, фигуры, бытовые детали, курьёзы, государственные решения
- Культурную историю: литература, искусство, музыка, кино, мода, язык — через призму "почему так получилось"
- Религиозную и церковную историю, историю текстов и канонов
- Городские странности, локальные феномены, историю мест (особенно Петербург)
- Истории государственных запретов, цензуры, ограничений — любых эпох и стран
- Антропологические и социологические сюжеты с историческим разворотом
- Истории об "ошибках истории" — переписывания, искажения, случайности
- Зарубежные исторические сюжеты с универсальным механизмом
- Видео других авторов на смежные темы (история, культура, антропология, религия, городские исследования)
- Видео, сильные по форме (монтаж, хук, звук, работа с архивом, подача рассказчика), если тема смежная — брать с пометкой "референс по форме"

**is_reference = false (не брать) если видео про:**
- Новостной/политический контент, политическая аналитика
- Личный лайфстайл: распорядок дня, рутины, гардероб, отношения
- Тревел в формате "10 мест, которые стоит посетить"
- Бизнес, маркетинг, саморазвитие, продуктивность
- Бьюти, мода, фитнес, кулинария (без историко-культурного разворота)
- Развлекательный контент: челленджи, трюки, pranks, скетчи без смысла
- Технологии, гаджеты, IT (если не история технологии как культурного феномена)
- Кино-обзоры в формате рецензии (без культурного разбора)
- Академические лекции, формат "учитель у доски"
- Узкие профессиональные ниши (медицина, юриспруденция, программирование)

**При сомнениях:** если тема смежная и можно извлечь хотя бы один работающий приём — is_reference = true. Если видео хорошо снято, но про полностью чужую тему (косметика, фитнес, стартапы) — is_reference = false.

**Реализация:** Полная версия правил выше встроена в системный промпт `backend/src/services/aiAnalysisService.js` (константа `BLOG_CONTEXT`) с Сессии 21. Если меняешь правила в этом разделе — синхронизируй и в коде.

## Дизайн — КРИТИЧЕСКИ ВАЖНО

### Общий подход
Сайт должен выглядеть профессионально и современно. НЕ как типичный vibe-coding проект.

### Конкретные требования
- **Шрифты**: НЕ Inter, НЕ Roboto, НЕ системные. Использовать характерные, качественные шрифты. Для заголовков — один выразительный шрифт, для текста — другой, читабельный. Подключать через Google Fonts.
- **Размеры элементов**: Крупные, удобные кнопки и элементы навигации. Никаких мелких кнопок. Clickable area минимум 44x44px. Текст — не мельче 14px, основной — 16px.
- **Цветовая схема**: Светлая тема. Не белый-на-белом — должна быть глубина и контраст. Акцентный цвет — один, яркий, используется точечно.
- **Пространство**: Щедрые отступы. Воздух между элементами. Таблица не должна выглядеть как Excel — строки с паддингами, hover-эффекты, читаемость.
- **Иконки**: Lucide React. Не перегружать, но использовать где улучшают навигацию.
- **Анимации**: Минимальные, но присутствуют — плавные переходы страниц, hover-эффекты, появление элементов.
- **Мобильная версия**: Полноценная, не урезанная. Меню — бургер. Таблица — карточки на мобильном.

### Эстетика
Ориентир — Notion, Linear, Raycast. Чистый, уверенный дизайн с характером. Не generic, не bootstrap-like.

- `backend/src/services/transcriptionService.js` — основной модуль:
  - **Извлекаем аудио, не отправляем сам mp4.** Whisper ограничен 25 МБ на файл, видео Reels уже на 30 сек могут весить 30–40 МБ. ffmpeg вытаскивает только звуковую дорожку: mp3, 64 kbps, моно, 16 kHz — речь читается отлично, размер падает в 20–50 раз. ffmpeg вызывается через `child_process.spawn` с бинарём из `ffmpeg-static` (нет системной зависимости, на Railway тоже работает). Все временные файлы — в `os.tmpdir()`, чистятся в `finally`
  - `downloadVideoToFile(url, path)` — потоково пишет mp4 на диск, без Referer (Instagram CDN отдаёт 403 на «чужой» Referer)
  - Whisper вызывается через `openai.audio.transcriptions.create({ file, model, language: "ru", response_format: "text" })` — `language: ru` это hint, не constraint, английская речь тоже распознаётся
  - Эвристика «нет речи»: после очистки от `[музыка]`, `♪`, пробелов остаётся <4 символов → `{ status: "no_speech", text: null }`. У Whisper нет явного флага «no speech», иначе пришлось бы детектить по постпроцессу
  - Возвращает `{ status: "done", text }` либо `{ status: "no_speech", text: null }`. Любая ошибка бросается наружу, ловит её процессор
- `backend/src/services/supabaseService.js` — добавлены `markTranscriptProcessing` / `saveTranscriptSuccess` / `saveTranscriptNoSpeech` / `saveTranscriptError`
- `backend/src/services/videoProcessor.js` — теперь два шага: сначала Apify (как раньше), потом транскрипция. Apify-ошибка → весь `processing_status = error` и стоп; ошибка транскрипции — изолированно в `transcript_status = error`, остальные данные остаются. Если Apify не вернул `videoUrl`, транскрипция помечается ошибкой с понятным текстом

**Сессия 12: Формула вирусности + исследование шеров** (завершена 2026-04-26)
- **Исследование шеров**:
  - `apify/instagram-scraper` (наш текущий актор, дефолт `APIFY_ACTOR_ID`) — `sharesCount` **не возвращает**. Подтверждено документацией актора + issue-трекером (тикеты «Scraping reels Shares» висят, не починены)
  - `apify/instagram-reel-scraper` — **возвращает `sharesCount`**. Это специализированный актор только под Reels, поля совместимы с нашими (`ownerUsername`, `caption`, `displayUrl`, `videoUrl`, `videoPlayCount`, `videoViewCount`, `likesCount`, `commentsCount`, `timestamp`) + дополнительно `sharesCount`
  - **Решение**: код умеет читать `sharesCount` из ответа любого актора. Дефолтный актор оставлен как был (`apify/instagram-scraper`) — переключение на `apify/instagram-reel-scraper` потребует тестирования, делать это не в рамках этой сессии. Если когда-нибудь понадобится — поменять `APIFY_ACTOR_ID=apify/instagram-reel-scraper` в `backend/.env`, никаких изменений в коде не нужно. - **Новая формула вирусности** в `backend/src/services/apifyService.js` → `computeVirality({ views, likes, comments, shares })`:
  - `((likes × 1) + (comments × 3) + (shares × 5)) / views × 100`
  - Веса: коммент ценнее лайка (порог входа выше — нужно реально что-то напечатать), шер ценнее коммента (распространяет в чужой контент). Если `shares = null` — слагаемое равно 0, формула продолжает работать
  - Старая формула `(likes + 2×comments) / views × 100` заменена. Существующие строки в БД пересчитаются только при повторной обработке (через retry или на новых видео); пересчитывать вручную через миграцию не стал — для текущего объёма данных у Влада не критично, и формулу мы ещё будем калибровать

**Сессия 15: Tinder-режим — каркас и свайп-механика** (завершена 2026-04-26)
- **Без `react-tinder-card`** — спецификация разрешала «или аналог», и я выбрал самописную реализацию через нативные pointer events. Причины: `react-tinder-card` (~2 года без обновлений, peerDeps на React 16/17) — риск несовместимости с React 19, который мы используем; и наш use case проще «колоды Tinder» — одна карточка за раз, своё мини-меню после свайпа влево, специфичные клавиши. Самописный код — ~330 строк в одном файле, меньше чем поверхность багов от внешней либы
- `frontend/src/lib/instagramEmbed.ts` — `extractInstagramShortcode(url)` достаёт shortcode из ссылок `/reel/`, `/reels/` и `/p/`. `buildInstagramEmbedUrl(url)` собирает `https://www.instagram.com/reel/<shortcode>/embed/` (минималистичный плеер без рекомендаций и панелей профиля). Регекс синхронизирован с `reelsUrlParser.ts`
- `frontend/src/components/blog/analyst/SwipeViewer.tsx` — полноэкранный режим (`fixed inset-0 z-[100]` через `createPortal` в body, чтобы не клипалось ничем выше по дереву):
  - **Снапшот списка**. Принимает `videos: Video[]` и фиксирует его через `useState(initialVideos)` на момент открытия. Удаления и переносы в закладки меняют parent state, но мы продолжаем итерацию по снимку — иначе индекс «прыгал» бы при каждой мутации
  - **Управление состоянием карточки**: `index` (текущая позиция), `drag: {dx, dy}` (активное перетаскивание), `exitDir: "left" | "right" | null` (карточка анимированно уезжает), `postLeftMenu: boolean` (после левого свайпа показано мини-меню), `busy` (пока async-операция). Каждое изменение `index` сбрасывает остальное через useEffect
  - **Pointer events** на самой карточке: `pointerdown` ловит начало (`setPointerCapture`, чтобы курсор за пределами не терялся), `pointermove` обновляет `drag`, `pointerup` решает: если `|dx| > 110px` или скорость >0.6 px/ms — фиксирует свайп, иначе снэп назад. Iframe исключён из перехвата (`if target.tagName === "IFRAME" return`) — чтобы Instagram-плеер получал свои клики; свайп работает по бортам карточки и инфо-панели снизу
  - **CSS-анимация ухода**: `transform: translateX(±1500px) rotate(±30deg); opacity: 0` с `transition: 250ms ease`. После `setTimeout(250)` — либо открываем `postLeftMenu`, либо инкрементим `index`
  - **Клавиатура**: глобальный `keydown`-listener — `←`/`→` триггерит свайп (с `e.preventDefault()`, чтобы не скроллить страницу), `Esc` закрывает viewer (или скрывает меню, если оно открыто)
  - **Карточка**: iframe с embed-URL внутри `flex-1` контейнера; `referrerPolicy="no-referrer"` обязателен (Instagram режет с чужим Referer). Если shortcode не извлечь — fallback на превью через `/api/thumbnail` + кнопка «Открыть в Instagram» по центру. Если ни URL, ни превью — короткое сообщение
  - **Инфо-панель** под видео: автор (uppercase, faint), AI-саммари (`line-clamp-2`), категория-pill через существующий `EntityChip` (цвет blue). Для категории `other` с подсказкой выводим как «Другое: <подсказка>» через готовый `formatAiCategory()`
  - **Layout**: header (close + счётчик «N из M» + spacer для центровки) → область карточки (карточка по центру + chevron-кнопки L/R справа и слева, скрыты на мобиле) → подсказка управления внизу
  - **Мини-меню после свайпа влево**: оверлей `bg-ink/60 backdrop-blur-sm` с диалогом, три кнопки — «🗑 Удалить» (красная), «📌 В закладки» (нейтральная), «Отмена» (текст). Выбор вызывает соответствующий коллбэк, при ошибке — alert и остаёмся
  - **Завершение списка**: когда `index >= total` — отдельный экран «Все видео просмотрены» с кнопкой возврата
- `AnalystWorkspace.tsx`:
  - State `viewerSnapshot: Video[] | null` — null = viewer закрыт
  - Кнопка «Режим просмотра» (`PlayCircle` icon) в тулбаре над таблицей рядом с fullscreen-кнопкой. Disabled при `filtered.length === 0`. Подпись «Режим просмотра» скрыта на мобиле (`hidden sm:inline`), остаётся только иконка
  - Открытие: `setViewerSnapshot(filtered)` — даём snapshot текущего отфильтрованного списка
  - `handleViewerDelete` и `handleViewerBookmark` — оптимистично режут видео из workspace state, при ошибке откатываются. Удаление в режиме просмотра идёт **без** ConfirmDialog (мини-меню само по себе подтверждение, диалог в диалоге — лишний клик); перенос в закладки использует существующий `svc.moveToBookmarks` (был готов с Сессии 14)

Решения, принятые намеренно:
- **Pointer events, не `react-tinder-card`.** Собственная реализация даёт контроль над поведением (мини-меню после свайпа, исключение iframe из drag-зоны), уменьшает количество зависимостей и не упирается в проблемы совместимости React 19. Не критично «иметь либу» — критично иметь нужное поведение
- **Снапшот списка на момент открытия.** Альтернатива — переживать live-обновления: но тогда удаление текущей карточки сразу подвинуло бы индекс и пользователь увидел бы скачок. Снимок гарантирует линейный обход
- **Iframe не перехватываем для свайпа.** Tinder-карточка с интерактивным контентом — вечный конфликт. Компромисс: свайп работает по бортам и инфо-панели, плюс есть видимые chevron-кнопки на десктопе и клавиатура. Мобильным — те же кнопки + жесты по бортам карточки (узкое место для UX, посмотрим как Влад пользуется и подкрутим если что)
- **Удаление без второго подтверждения.** Мини-меню «Удалить / В закладки / Отмена» уже выполняет роль подтверждения — навешивать сверху ConfirmDialog бессмысленно
- **Свайп вправо в этой сессии — просто переход.** Воронка с категорией/заметкой/тегами — это Сессия 16. Сейчас правый свайп листает дальше, не теряя данные
- **Кнопка в тулбаре workspace, не в шапке страницы.** Спецификация писала «рядом с Добавить видео», но кнопке нужно знать `filtered.length` (для disabled), а это live state клиентского workspace. Класть state-зависимую логику в server-компонент через прокидывание — неоправданно сложно. В тулбаре рядом с fullscreen — естественное место для «view-controls»

- Изменения в `SwipeViewer.tsx`:
  - Новые пропы: `myCategories`, `tags`, `onCreateMyCategory`, `onCreateTag`, `onSubmitRightFlow`

  - **Правый свайп НЕ запускает анимацию ухода**: `triggerSwipe("right")` теперь только сбрасывает drag и ставит `rightFlowOpen=true`. Карточка остаётся на месте — видна за полупрозрачным `bg-ink/70` подложки воронки. Это явное требование спеки «видео остаётся видно на фоне, затемнённое»

Решения, принятые намеренно:
- **Без анимации ухода на правом свайпе.** Альтернатива — анимировать карточку в сторону, потом показать модалку. Но спека прямо говорит: видео должно быть видно за затемнённой подложкой. Это естественно для «приостановки на воронке» — пользователь видит, что ещё работает с тем же видео, не успел потерять контекст
- **Без `EntitySelectPopover` в флоу.** Поповер требует якорь и открывается по клику — для шага «выбери категорию» это лишний клик. Инлайн-список с тем же поведением (поиск + создание + клик-выбор) даёт single-click flow
- **Сохранение одним батчем в самом конце, не пошагово.** Альтернатива — сохранять каждый шаг отдельно. Это чище для отмены: если пользователь нажмёт «Закрыть» на любом шаге, в БД ничего не записано. Один atomicity-юнит на флоу
- **Создание тегов внутри флоу разрешено.** Спека буквально пишет «ряд тег-кнопок (все существующие теги)», но на практике пользователь часто хочет добавить новый ярлык на лету. Мини-инпут «Новый тег» снизу — минимально навязчивый аффорданс, использовать необязательно
- **«Пропустить» на шаге оценки сохраняет verified.** «Пропустить» — это «не хочу думать про градацию», а свайп вправо уже выразил «одобрено». Без этого пользователь, который хочет быстро пройти весь флоу одним пропуском, не получит даже базовой оценки

Известные ограничения сессии 16:
- Если запрос упадёт прямо во время сохранения — alert с ошибкой, карточка не листается, локальный state откатывается. Пользователь может закрыть воронку и попробовать в табличном режиме. Retry прямо из воронки не предусмотрел — для пятисекундного флоу это лишний клик
- В шаге 2 при создании тега новый тег попадает в state `tags` (через `handleCreateTag` в workspace) и автоматически выбирается. Если параллельно открыта другая вкладка с тем же приложением — там новый тег появится только при следующем revalidatePath, не realtime

**Сессия 16.1: Multi-select для оценок** (завершена 2026-04-27)
- **Исправляем поведение из Сессии 16**: оценка перестаёт быть single-select, теперь это массив. Влад может одновременно отметить, например, ✅ и 🔥 — это два разных аспекта (полезно vs выдающееся), и хранить только одно — потеря смысла
- Миграция `supabase/migrations/0008_ratings_multiselect.sql`:
  - Добавляет колонку `videos.ratings text[] not null default '{}'`
  - Переносит существующие значения: `update set ratings = array[rating] where rating is not null` (старые одиночные оценки превращаются в массив из одного элемента — данные не теряются)
  - CHECK `ratings <@ array['verified','super','repeat']` — каждое значение из допустимого набора, оператор `<@` это «содержится в»
  - GIN-индекс по массиву (`using gin (ratings)`) — быстрый фильтр «оценки содержат X»
  - Удаляет старую колонку `rating` (вместе с её индексом и check-constraint автоматически)
- Тип `Video.rating: Rating | null` → `Video.ratings: Rating[]`. Пустой массив = «не оценено». Параллельно в `videosService.ts` маппинг строки БД использует `isRating()` type-guard — если в БД случайно попадёт мусор (например, после ручного вмешательства), отфильтруется молча
- `manualFieldsService`: `setVideoRating(id, rating)` → `setVideoRatings(id, ratings)`. `SwipeRightPayload.rating?: Rating | null` → `SwipeRightPayload.ratings?: Rating[]`. Семантика остаётся: `undefined` — не трогать, заполненный массив — записать
- `videoFilters.matchesRating`: было `v.rating === filter`, стало `v.ratings.includes(filter)`. Фильтр «Не оценено» — `v.ratings.length === 0`. UI фильтра в `FilterBar` без изменений (тип `RatingFilter` тот же)
- `RatingCell.tsx` переписан в multi-select:
  - Кнопка-якорь показывает все эмодзи в ряд (был один). При двух-трёх оценках кнопка чуть шире (`px-1.5` вместо квадратной)
  - Поповер: каждый пункт — toggle (`aria-pressed`), галочка для активных. **Поповер не закрывается на toggle** — можно отметить сразу два-три значения одним заходом, как в `TagsCell`
  - Кнопка «Снять все оценки» снизу, активна только когда есть хоть что-то
- `SwipeRightFlow.RatingStep` тоже multi-select: state `ratings: Rating[]` стартует с `["verified"]` (свайп вправо = одобрено по умолчанию), кнопки ✅/🔥/🔄 работают как toggle. «Готово» сохраняет state как есть; «Пропустить» — с дефолтом `["verified"]`. Текст подсказки переписан под новое поведение
- Имена коллбэков переименованы для ясности: `onSelectRating` → `onUpdateRatings` (`AnalystWorkspace`, `VideoTableRowCallbacks`, `ThumbnailCell`). Меняли семантику — лучше менять и имя, чтобы старое значение больше не путало

Решения, принятые намеренно:
- **`text[]` в БД, не три boolean-колонки.** Массив гибче (если вдруг добавим ✨ или ⭐ — миграция дешевле, в отличие от трёх отдельных полей). CHECK ограничивает значения текущим набором
- **Не пишем `null` ни в payload, ни в БД.** Семантика «не оценено» = пустой массив. Это снимает вопрос «это `null` потому что юзер не трогал, или потому что снял?» — `undefined` означает «не трогали», `[]` — «снял всё». Чище для diff'ов и для отладки
- **GIN-индекс без частичного `WHERE`.** Прошлый `videos_rating_idx` был частичным (`WHERE rating IS NOT NULL`) — разумно для одиночного значения. Для массива нет смысла исключать пустые: GIN сам умеет с этим работать, частичный индекс усложнит query plan'ы
- **Существующие записи мигрируют автоматически.** Те, у кого `rating` стоял — получат массив из одного элемента. Те, у кого `null` — получат пустой массив. Никаких ручных правок Влад делать не должен

1. **Миграция Supabase**: Dashboard → SQL Editor → вставить целиком `supabase/migrations/0008_ratings_multiselect.sql` → Run. ⚠️ **Эта миграция дропает колонку `rating`** в `videos` (заменяя её на `ratings`). Существующие оценки переносятся автоматически в первый шаг up-перед drop, **но если ты вручную писал что-то нестандартное в `rating` через Dashboard — сделай бэкап**. - **`is_reference` хранится в `videos`, не в отдельной таблице.** Это атрибут видео, связь 1:1 — отдельная таблица была бы избыточна. CHECK не нужен, потому что boolean ограничивает значения сам
- **Toggle через клик, без поповера и без меню.** Альтернатива — поповер с тремя кнопками (✅/❌/снять). Но пользователь обычно знает, что хочет, и поповер добавляет лишний клик. Циклический toggle с подсказкой в `title` нагляднее

Решения, принятые намеренно:
- **Cleanup-режим в том же `SwipeViewer.tsx`, не отдельный компонент.** 95% поведения общее (карточка, плеер, свайпы, клавиатура, медиа-fallback) — копировать всё в `CleanupViewer.tsx` означает поддерживать две похожие реализации. `mode`-флаг разруливает 5% различий в `triggerSwipe` и заголовке.
- **Анимация ухода вправо в cleanup, но не в default.** Это намеренная асимметрия. В default правый свайп открывает воронку поверх карточки — там важно видеть видео сквозь полупрозрачную подложку, поэтому карточка остаётся на месте. В cleanup решение принимается жестом сразу, и пользователю важно увидеть «карточка улетела вправо = в основной базе» и «улетела влево = в закладки» — это ментально подтверждает выбор.
- **«Очистка» работает по всем видео, а не по `filtered`.** Альтернативный вариант (только по фильтру) сделал бы кнопку зависимой от текущего state фильтров — пользователь может удивиться, что в режиме нет всех видео. Спецификация писала «автоматически показывает видео с `is_reference=false`» — без оговорок про фильтры.
- **Bulk-перенос через `moveManyToBookmarks`, последовательно, не batch-RPC.** Для нескольких десятков видео это ~3–10 секунд, что для одноразовой ручной операции приемлемо. Batch-RPC с транзакцией требовал бы хранимой функции на стороне Supabase — это +миграция и +поддержка ради экзотической оптимизации.
- **При ошибке в bulk-bookmark откатываем весь снапшот.** Альтернатива — отслеживать, какие id уже успели уехать, и держать их удалёнными. Это сложнее и не помогает: при revalidatePath реальная картина подтянется автоматически, а пока пользователь видит «лишние» строки → видит, что часть осталась, и может попробовать ещё раз.

Известные ограничения сессии 19:
- Cleanup-режим использует **снимок видео на момент открытия** (как и default-режим). Если AI-анализ в фоне для какого-то видео завершится прямо в момент работы пользователя в cleanup и пометит ещё одно как `is_reference=false` — оно не попадёт в текущий заход. После закрытия кнопки `cleanupCount` обновится, и можно будет открыть снова.
- Если bulk-bookmark упал на середине пачки — часть видео уехала, часть нет. Snapshot восстанавливает все строки, но в БД часть уже в bookmarks. Это будет видно после revalidatePath. Промежуточное состояние не идеально, но в худшем случае пользователь повторно нажмёт «В закладки», и оставшиеся доедут (`upsert(onConflict: "url", ignoreDuplicates: true)` в `moveToBookmarks` защищает от дублей).
- Свайп вправо в cleanup использует **тот же `EMBED_TIMEOUT_MS = 5000`** — для чистого видео это норма, но если у пользователя медленный интернет и он быстро свайпает, embed может не успеть загрузиться даже до полного ухода карточки. На практике плеер не нужен для решения «для блога / другое» (по описанию + саммари обычно ясно), так что это не блокер.

Ручные шаги для Влада (после pull):
1. **Перезапускать ничего не нужно** — это чисто фронтовая сессия без миграций и новых зависимостей. Hot-reload Next.js подтянет всё.
2. На странице «База референсов» в плавающей панели (внизу, когда выбраны строки) появилась кнопка «📁 В закладки» рядом с «Удалить». Сценарий: фильтр «Референс = ❌ Другое» → выделил всё → «В закладки» → подтверждение → видео уезжают.
3. В тулбаре над таблицей рядом с «Режим просмотра» появилась кнопка «Очистка» — со счётчиком сколько ещё разобрать (например, «Очистка · 47»). Активна только если AI пометил хотя бы одно видео как «не для блога».
4. Клик по «Очистка» → полноэкранный режим: одно видео по центру, заголовок «Осталось проверить: N». Свайп влево (мышью / клавишей `←` / drag) → видео уходит в закладки. Свайп вправо → помечается как «для блога», остаётся в базе. Без меню, без воронки — просто бинарный выбор за два жеста.
5. Когда все видео разобраны → экран «Очистка завершена» с кнопкой возврата к таблице.

**Сессия 20: Постоянные превью через Supabase Storage** (завершена 2026-05-02; первоначально планировалась через Google Drive, но переехали на Supabase — см. ниже)
- Решение проблемы: ссылки Instagram CDN на превью протухают за ~24 часа. Теперь после парсинга бэкенд скачивает превью и заливает его в публичный bucket Supabase Storage — постоянный URL и относительный path сохраняются в БД, при удалении видео файл из bucket'а подчищается.

История: сначала было реализовано через Google Drive Service Account, но в проде Drive отказал с ошибкой `Service Accounts do not have storage quota` — Service Account-ы Google **не имеют собственной квоты** и могут писать только в Shared Drive (доступен только в Google Workspace, ~$6/мес). Переехали на Supabase Storage: уже есть в проекте, бесплатно до 1 ГБ места и 2 ГБ исходящего трафика в месяц, ключ доступа (`SUPABASE_SERVICE_ROLE_KEY`) уже прокинут — никаких новых сервисов и аккаунтов.

- Миграции:
  - `supabase/migrations/0010_thumbnail_drive_id.sql` — добавила колонку `thumbnail_drive_id` (legacy от Drive-плана). Миграция уже выполнена в проде, поэтому не откатываем
  - `supabase/migrations/0011_thumbnail_storage_path.sql` — переименовывает `thumbnail_drive_id` → `thumbnail_storage_path` в `videos` и `bookmarks`. Реальных данных в колонке не было (Drive-загрузка ни разу не сработала), переименование безопасное

  - `backend/src/services/supabaseStorageService.js` — клиент Supabase Storage:
    - Авторизация через `SUPABASE_SERVICE_ROLE_KEY` (тот же, что для writeable-операций по БД), service-role клиент минует RLS bucket'а
    - `uploadThumbnail(imageUrl, filename)` — fetch картинки без Referer → `supabase.storage.from(bucket).upload(...)` с `upsert: true` → возвращает `{ url, path }`. URL — публичный CDN-URL Supabase, встраиваемый в `<img>` напрямую без прокси
    - `deleteThumbnail(path)` / `deleteManyThumbnails(paths)` — Supabase API принимает массив путей одним запросом, отсутствующие файлы не считает ошибкой
    - **`isEnabled()` возвращает true когда `SUPABASE_STORAGE_BUCKET` задан**. Логирует warning ровно один раз (флаг `warnedAboutDisabled`), чтобы на массовом батче не зашумлять лог
  - `backend/src/config/env.js` — одна новая переменная `storageBucket` (опциональная). Если не задана — бэкенд работает как раньше, превью остаются на Instagram CDN
  - `backend/src/services/videoProcessor.js` — после Apify, **до** `saveVideoSuccess`: если есть `thumbnail_url` и Storage включён, пытаемся залить. При успехе — заменяем `fields.thumbnail_url` на CDN-URL Supabase и кладём `fields.thumbnail_storage_path`. При ошибке — оставляем оригинальный URL, продолжаем (видео не блокируется). Имя файла: `<shortcode>-<timestamp>.jpg`

Решения, принятые намеренно:
- **Supabase Storage вместо Google Drive.** Service Account на бесплатном Google не может писать на Drive. Workspace ($6/мес) или OAuth (своя квота) решают, но добавляют сложность и/или стоимость. Storage уже есть, бесплатный лимит покрывает все обозримые объёмы для личного инструмента
- **`SUPABASE_STORAGE_BUCKET` опциональная.** Если переменная не задана — бэкенд продолжает работать как до Сессии 20: превью с Instagram CDN, протухают через сутки. Это критично на этапе деплоя: миграция БД может пройти раньше, чем создан bucket, и в этот момент система не должна падать
- **Залить превью в Storage — лучшее усилие, не блокер.** Любая ошибка (Storage недоступен, картинка не скачалась) → логируем warning, оставляем оригинальный URL Instagram, идём дальше. Видео обрабатывается успешно
- **Public bucket, не private с signed URLs.** Превью — не секрет: Instagram-посты публичные, и наша БД в `videos` тоже хранит публичные ссылки. Public bucket даёт прямой CDN-URL, не нужны сервер-генерируемые ссылки на каждый просмотр, проще встраивать в `<img>`
- **Прокси `/api/thumbnail` оставлен, но для Supabase URL не критичен.** Supabase CDN отдаёт картинки с правильными CORS-заголовками, прокси нужен только для старых записей с Instagram CDN (которые ещё не мигрированы). Postoянный whitelist даёт чистоту: всё проходит через один путь, кэширование Vercel CDN на сутки работает для обоих источников
- **Имя файла = `<shortcode>-<timestamp>.jpg`.** Уникальный timestamp защищает от конфликтов на retry
- **Bookmarks тоже получают `thumbnail_storage_path`.** Колонка добавлена в обе таблицы — при будущем UI закладок и переносе превью в `moveToBookmarks` поле уже доступно

Известные ограничения сессии 20:
- **Бесплатный лимит Supabase: 1 ГБ места и 2 ГБ трафика в месяц.** При типичном превью ~50 КБ это ~20 000 файлов. Трафик считается на каждом скачивании браузером — открытие страницы с таблицей грузит ~50 миниатюр × 50 КБ = ~2.5 МБ. То есть ~800 открытий страницы в месяц без апгрейда. Для личного инструмента — с большим запасом
- **Bookmarks → удаление:** удаление из bookmarks (через Supabase Dashboard, UI пока нет) **не подчищает Storage**. Файлы останутся «сиротами». Когда появится UI закладок — добавим тот же flow, что для видео
- **Скрипт миграции старых превью.** `POST /api/thumbnails/migrate` — синхронный, обрабатывает 50 видео за вызов (~20–40 секунд). Запускать вручную через DevTools fetch. Для нескольких сотен видео — кликнуть несколько раз

**Сессия 21: Обновление AI-промпта с полным контекстом блога** (завершена 2026-05-02)
- Миграций БД нет — только бэкендовая логика. Фронт не трогали, env-переменные не меняются.
- `backend/src/services/aiAnalysisService.js`:

Тематика блога (история, культура, точки поворота, Парфёнов-стиль), полные списки «что брать / что не брать», правила при сомнениях. То есть на пограничных случаях AI чаще будет ставить true
  - Добавлен узкий `RECLASSIFY_SYSTEM_PROMPT` — только классификация `is_reference`, без саммари и категории. Используется одноразовым endpoint'ом для переклассификации старых видео; экономит токены вывода (не генерирует ненужные саммари заново)
  - Новая функция `reclassifyIsReference({ caption, transcript, summary })` — отдельный путь только для пересчёта `is_reference`. Передаём прежнее AI-саммари как дополнительный контекст: оно компактное, и часто это лучший сигнал для классификации. `temperature: 0.2` (чуть жёстче, чем 0.3 в основном анализе) — на повторных прогонах нужна стабильность. Возвращает `{ status: "done", isReference: boolean | null }` либо `{ status: "skipped" }`
- `backend/src/services/supabaseService.js`:
  - `getVideosForReferenceReprocessing({ limit, before })` — выборка для пачки. Фильтр: `content_type = 'video'` (для image/carousel мы принудительно ставим false ещё в `videoProcessor.js` — повторный AI-вызов бесполезен) и `ai_summary IS NOT NULL` (значит данные точно есть). Сортировка `created_at desc` + keyset-пагинация через `before` (lt по `created_at`) — повторные вызовы не возвращают тех же самых
  - `setVideoIsReference(id, value)` — точечный апдейт колонки, не трогая саммари/категорию/статус
- `backend/src/routes/reprocess.js` (новый):
  - `POST /api/reprocess-references` принимает `{ limit?: number (default 50, max 200), before?: ISO-строка }`. Проходит по выбранной пачке последовательно (между видео — пауза 200мс на всякий случай против OpenAI rate-limit), для каждого вызывает `reclassifyIsReference`, при изменении значения — `setVideoIsReference`
  - Ответ: `{ requested, processed, changed, unchanged, skipped, failed, nextCursor, remainingHint, changes[], errors[] }`. `changes` (до 50 записей) — список `{id, from, to}`, чтобы Влад мог посмотреть, что именно поменялось; `errors` — до 10 ошибок с укороченным сообщением
  - Когда `pending.length < limit` — значит дошли до конца, `nextCursor = null`, в `remainingHint` подсказка
- `backend/src/app.js` — роутер подключён без префикса (`app.use("/", reprocessRouter)`), внутри роутера сам путь `/api/reprocess-references` — чтобы получить тот URL, который зафиксирован в спеке Сессии 21

Решения, принятые намеренно:
- **Отдельная функция `reclassifyIsReference`, а не переиспользование `analyzeVideo`.** Альтернатива — прогнать `analyzeVideo` повторно и взять только `isReference`. Но тогда модель тратит токены на регенерацию саммари+категории, которые мы не записываем; на 1000 видео это лишние ~$0.5–1 и ~30% времени. Узкая функция со своим промптом дешевле и быстрее
- **Передаём AI прежнее саммари как часть контекста.** Caption бывает скудный, транскрипция — тоже не всегда. Уже сгенерированное саммари компактно ухватывает суть, и для одной задачи «решить про блог / не блог» этого достаточно. Если саммари у видео нет (мало вероятно, фильтр `ai_summary IS NOT NULL` это исключает) — функция всё равно работает на caption + transcript
- **Keyset pagination через `before` курсор, а не offset.** Offset небезопасен при параллельных мутациях (если пользователь добавит видео между вызовами — пачки сместятся, кого-то пропустим). Курсор по `created_at` стабилен
- **`temperature: 0.2` против 0.3 в основном промпте.** В `analyzeVideo` нужна свежесть формулировок саммари — небольшое разнообразие полезно. В `reclassifyIsReference` нет генерации текста, только бинарная классификация — стабильность важнее
- **Не пересчитываем для image/carousel.** Бесполезно: их `is_reference` принудительно ставится в false на этапе processor'а; даже если AI ответит true, мы перезапишем в false при следующем retry. Чтобы не плодить путаницу — проще вообще не вызывать модель на них
- **Endpoint синхронный, обрабатывает limit за вызов.** Альтернатива — длинный фоновой процесс на бэкенде с прогрессом в БД. Для одноразовой операции с keyset pagination это перебор: Влад запустит несколько раз подряд из DevTools, между вызовами увидит сводку, поймёт, что меняется, остановится если что не так. Контроль выше — сложность ниже
- **Эндпоинт без CSRF/auth защиты.** Бэкенд изначально не имеет аутентификации (RLS открытая на anon во всех таблицах) — это согласовано как «личный инструмент для одного пользователя» в более ранних сессиях. Reprocess в той же модели

Точечный maxWidth на ячейках не ломает существующую логику layout, ограничивая только то, что должно быть ограничено
- **Свёрнутая шапка — chevron-only, без короткой подписи**. - **`Shift+←` для удаления, не `Delete` или другая клавиша**. `Shift+←` логически связано с обычным `←` («в закладки»): это «то же действие, но более деструктивное». Семантическая близость в раскладке клавиш помогает запомнить: одна рука работает с левой стрелкой, добавляя/убирая `Shift`

   - **Клавиатура**:
     - `←` → видео в закладки

     - `→` → оставить (в обычном режиме откроется воронка категория/теги/оценка; в режиме «Очистка» сразу пометит как «для блога»)

   - Подсказка с раскладкой висит внизу экрана, чтобы не забыть

**Сессия 24: Раздел «Команда» — миграция БД и Storage buckets** (завершена 2026-05-04)

Старт нового большого блока работ — реализация раздела **Блог → Команда** внутри Потока. Это перенос локальной десктопной утилиты «ДК Лурье» (инструмент Влада для подготовки экскурсий с LLM-задачами и журналом расходов) в облако Потока, с расчётом на этапе 2 эволюционировать в систему AI-агентов для блога. Архитектура и план — в файлах `STAGE1_ARCHITECTURE_v2.md` и `ROADMAP_STAGE1_v2.md` в корне репо.

**Эта сессия — Сессия 1 из 8 в roadmap-е этапа 1 раздела «Команда»**, в нумерации сессий Потока — Сессия 24.

Зачем нужен раздел «Команда»: в локальной ДК Лурье у Влада есть инструмент с пятью типами задач LLM (свободные идеи, идеи под исследование, прямое исследование URL/PDF, написание текста, AI-правка фрагментов), своими шаблонами промптов, журналом расходов и хранилищем артефактов. Этап 1 — полностью повторить эту функциональность в Потоке (на стеке Node.js + Next.js + Supabase Потока, без переписывания на Python). Этап 2 (после) — превратить инструмент в команду AI-агентов с идентичностью, расписаниями и памятью.

В этой сессии трогаем только БД и хранилище — никакого фронта, никакого бэкенда. Чисто инфраструктурный шаг.

- Миграция `supabase/migrations/0012_team_tables.sql` создаёт четыре таблицы и три bucket'а в Storage. Все имена с префиксом `team_` / `team-`, чтобы новые сущности не путались с существующими `videos`, `bookmarks`, `thumbnails` Потока.

- **Таблица `team_tasks`** — журнал задач команды, append-only:
  - `record_id` (uuid pk) — id записи (уникальный для каждой строки)
  - `id` (text) — id задачи (одинаковый для всех снапшотов одной задачи; генерируется бэкендом, формат `tsk_<random>`)
  - `type` / `title` / `status` (running / done / revision / archived / error / marked_done)
  - `params` (jsonb) — входные параметры задачи (user_input, source, point_name, length_hint, research_paths и т.п.)
  - `model_choice` (jsonb) — пресет / провайдер / модель, выбранные пользователем
  - `provider` / `model` (text) — резолвлёные после `resolveModelChoice`
  - `prompt` (jsonb) — `{system, user, cacheable_blocks, template}`
  - `prompt_override_used` (boolean) — был ли промпт отредактирован пользователем перед запуском
  - `result` (text) — финальный текст ответа LLM (дублирует артефакт для быстрого превью)
  - `artifact_path` (text) — путь в Storage до артефакта (относительно bucket'а `team-database`)
  - `tokens` (jsonb) — `{input, output, cached}`
  - `cost_usd` (numeric)
  - `error` (text)
  - `created_at` / `updated_at` / `started_at` / `finished_at` (timestamptz)
  - Индексы: основной — `(id, created_at desc)` под паттерн чтения `SELECT DISTINCT ON (id) ... ORDER BY id, created_at DESC` (текущее состояние задачи = последняя строка с этим id); сортировка списка — `created_at desc`; частичный `where status = 'running'` для recovery после рестарта и для UI «В процессе»
  - **CHECK на `type` намеренно не ставится** — на этапе 2 список типов расширяется (новые типы под агентов), валидация в коде снимет необходимость миграции на каждый новый тип
  - Триггер `set_updated_at()` (функция уже создана миграцией 0001) — append-only это соглашение, но защититься от случайных UPDATE на уровне БД дёшево

- **Таблица `team_api_calls`** — журнал вызовов LLM для UI «Расходы» и для биллинга AI-правок против родительской задачи через `task_id`:
  - `id` (uuid pk), `timestamp` (timestamptz)
  - `provider` / `model` (text)
  - `input_tokens` / `output_tokens` / `cached_tokens` (int) + `cost_usd` (numeric)
  - `task_id` (text, nullable) — ссылка на task id из `team_tasks`. Nullable для вызовов вне контекста задачи (например, refinePrompt)
  - `success` (boolean) / `error` (text) — для отладки и метрик
  - `audio_minutes` (numeric) — для Whisper-вызовов, биллинг по минутам, не по токенам
  - Индексы: `timestamp desc` (журнал) + частичный `task_id where task_id is not null` (биллинг по задаче)

- **Таблица `team_api_keys`** — ключи моделей в БД, не в env:
  - `provider` (text pk, CHECK in `'anthropic'|'openai'|'google'`) + `key_value` (text) + `updated_at`
  - В env Railway хранить ключи плохо — каждое изменение требует передеплоя. Здесь Влад через UI «Админка» меняет ключ, бэкенд через `keysService` подхватывает с 30-сек кешем
  - **Шифрования на уровне БД пока нет** — Поток сейчас тоже хранит `OPENAI_API_KEY` и `APIFY_API_TOKEN` в env Railway без шифрования; модель безопасности «личный инструмент без auth» одинаковая. Если в будущем понадобится — добавим через `pgsodium`

- **Таблица `team_settings`** — гибкий key-value для настроек команды (минимально — `alert_threshold_usd` для алерта расходов): `key` (text pk) + `value` (jsonb) + `updated_at`. Гибкий формат, чтобы добавлять новые настройки без миграций

- **RLS на всех четырёх таблицах открытая** (политика `team_*_public_all` на all для anon/authenticated), как у `videos` и `bookmarks` — приложение пока без auth. Service-role клиент бэкенда обходит RLS, browser-клиент ходит через anon

- **Storage buckets** (через `insert into storage.buckets ... on conflict do nothing` в той же миграции):
  - `team-database` — артефакты задач: `research/`, `texts/<point>/`, `ideas/`, `sources/`, `uploads/`, плюс `context.md` и `concept.md` в корне
  - `team-prompts` — 5 markdown-файлов шаблонов промптов
  - `team-config` — `pricing.json` (тарифы моделей) и `presets.json` (пресеты выбора модели)
  - **Все три приватные** (`public = false`): доступ через service-role бэкенда. Браузер напрямую не читает — все операции с артефактами/промптами идут через `backend/src/routes/team/artifacts.js` и `/prompts.js` (см. Сессию 27 / 4 в roadmap-е команды)
  - Если SQL-инициализация bucket'ов упадёт (зависит от прав клиента, через который запускается миграция) — в комментарии миграции есть инструкция создать руками в Dashboard → Storage → New bucket

Решения, принятые намеренно:
- **Номер миграции `0012`, а не `0008` как в архитектурном документе.** Документ писался под состояние, когда последняя миграция Потока была `0007`. С тех пор Поток добавил `0008`–`0011` (Сессии 16.1, 18, 20). Принцип «продолжаем нумерацию Потока» сильнее буквы документа — присвоен следующий свободный номер
- **Сессия в `CLAUDE.md` помечена как Сессия 24, а не Сессия 17 / «Команда — Сессия 1».** В стартовом задании было сказано «продолжая нумерацию Потока», и `CLAUDE.md` уже содержит Сессии 17–23. Чтобы поиск по «Сессия N» в этом файле оставался однозначным, продолжаем общую нумерацию. Внутри текста сессии явно указана и параллельная нумерация по roadmap-у команды («Сессия 1 из 8 в roadmap-е этапа 1»), чтобы было удобно сверяться с roadmap-ом
- **Префикс `team_` / `team-` обязателен.** Без него `tasks` / `api_calls` / `settings` могли бы конфликтовать с будущими таблицами других разделов суперапа (Финансы, Новости и т.д., которые у Потока запланированы в будущем)
- **`type` в `team_tasks` без CHECK constraint.** Этап 2 добавит новые типы под AI-агентов (например, `agent_self_review`, `telegram_dispatch`). CHECK потребовал бы миграции на каждое расширение; валидация в коде через TASK_HANDLERS-реестр даёт ту же гарантию без миграций
- **`record_id` PK + отдельный текстовый `id` без unique.** Append-only — каждая строка с тем же `id` означает новый снапшот состояния задачи. UNIQUE на `id` сломал бы паттерн. PK на `record_id` нужен для делитов конкретного снапшота (на этапе 1 не используется, но архитектурно корректно)
- **`provider` и `model` дублируются в `team_tasks` и в `team_api_calls`.** В `team_api_calls` они нужны для агрегации расходов по моделям (UI «Админка → Расходы по моделям») без обязательного джойна на задачу. В `team_tasks` — чтобы запись задачи была самодостаточной для UI «карточка задачи» (модель показывается в карточке, нет смысла джойнить api_calls для отображения метаданных). Денормализация оправдана UX-сценариями
- **Buckets приватные, не публичные.** В отличие от существующего `thumbnails` (public) для превью видео — артефакты команды (черновики текстов, исследования, контекст блога) приватнее по природе. Service-role-only access усиливает изоляцию: даже если злоумышленник получит anon-ключ, он не сможет вытащить артефакты. Бэкенд проверяет path и проксирует чтение в Сессии 27 (Сессии 4 roadmap-а команды)

Известные ограничения сессии 24:
- Таблицы и bucket'ы созданы, но **в системе нет ни одного приложенного потребителя** — бэкенд и фронт не знают про эти таблицы, никаких UI-изменений нет. Это нормально: это первая из 8 сессий этапа, дальнейшие сессии добавят сервисы (Сессия 25), TaskRunner и handlers (Сессия 26), API-маршруты (Сессия 27), фронт (Сессии 28–30) и финальную полировку (Сессия 31)

- Если SQL-инициализация bucket'ов в Supabase Dashboard не сработает (например, из-за прав на схему `storage`) — нужно создать bucket'ы вручную через UI. Инструкция в комментарии миграции

**Сессия 25: Команда — бэкенд-ядро сервисов (порт ДК Лурье)** (завершена 2026-05-04)

**Эта сессия — Сессия 2 из 8 в roadmap-е этапа 1 раздела «Команда»**, в нумерации сессий Потока — Сессия 25. Базируется на инфраструктуре, поднятой в Сессии 24 (таблицы `team_*` и bucket'ы `team-*` в Supabase).

Эта сессия — портирование с Python на Node.js трёх ключевых сервисов команды из локальной ДК Лурье плюс инфраструктурные клиенты Supabase для team_* таблиц и team-* buckets. Без задач пока, только инструменты.

Локальные референсы (Python-исходники из ДК Лурье) лежат в `_reference_dkl/` в корне репо — папка добавлена в `.gitignore`, в Git не попадает.

- **Зависимости** (`backend/package.json`): `@anthropic-ai/sdk` (^0.93) и `@google/generative-ai` (^0.24). OpenAI SDK уже был для Whisper/анализа Потока — переиспользуем тот же
- **Новая папка** `backend/src/services/team/` с шестью файлами

- **`teamSupabase.js`** — клиент Supabase для всех team_* таблиц, единственный singleton service-role клиента в этом разделе:
  - `appendTaskSnapshot(snapshot)` — append-only паттерн из ДК Лурье. На каждое изменение состояния задачи — новая строка в `team_tasks` с тем же `id`, текущее = последний снапшот. Поля camelCase из JS превращает в snake_case БД (`modelChoice` → `model_choice`, `artifactPath` → `artifact_path` и т. д.), `costUsd` маппится в `cost_usd`
  - `getTaskById(id)` — последний снапшот по id
  - `getActiveTaskIds()` — id всех задач, у которых **последний** снапшот = status `running`. Реализация через клиентский DISTINCT ON: тащим всё, сортированное по `created_at desc`, проходим первый раз каждый id. Для команды это десятки строк, не нагрузка
  - `getAllTasks()`, `getTasksByStatus(status|status[])` — текущие состояния всех задач
  - `recordApiCall(payload)` — пишет в `team_api_calls`. **Не бросает на ошибке** — журнал не должен валить основной поток LLM-вызова, ошибку логируем и идём дальше
  - `getApiCallsByTaskId(taskId)`, `getAllApiCalls()` — для агрегации расходов
  - `getSetting(key)`, `setSetting(key, value)` — `team_settings` (jsonb), используется для `alert_threshold_usd` и потенциально для других настроек

- **`teamStorage.js`** — клиент трёх buckets команды (team-database / team-prompts / team-config):
  - `uploadFile(bucket, path, content)` — `content` принимает строку или Buffer. Авто-определение `content-type` по расширению (md/json/txt/html/pdf), `upsert: true` (повторная заливка перезаписывает). Использовалось бы при сохранении артефактов задач в Сессии 26 (Сессия 3 roadmap'а команды)
  - `downloadFile(bucket, path)` — возвращает контент как UTF-8 строку (Storage SDK на Node 20+ возвращает Blob, конвертируем через `.text()`)
  - `fileExists(bucket, path)` — true/false без бросков, для проверок «загружать или нет»
  - `listFiles(bucket, prefix)` — `limit: 1000`, сортировка по имени для стабильного UI
  - `deleteFile(bucket, path)` — best effort, ошибки логирует

- **`keysService.js`** — управление ключами в таблице `team_api_keys`. Решение хранить в БД, а не в env, задокументировано в `STAGE1_ARCHITECTURE_v2.md` (раздел 3.3) — Влад сможет менять ключи через UI Админки без передеплоя:
  - `getApiKey(provider)` с **кешем 30 секунд** в памяти процесса. Компромисс: не дёргаем БД на каждом LLM-вызове, но при смене ключа в Админке свежее значение подтянется максимум через 30 сек
  - `setApiKey(provider, key)` / `deleteApiKey(provider)` — чистят кеш (следующий getApiKey прочитает из БД)
  - `getAllKeysStatus()` — возвращает `{anthropic: bool, openai: bool, google: bool}` для UI «🟢 / 🔴» в Админке
  - Поддерживаемые провайдеры — белый список (`anthropic | openai | google`), любой другой бросает «Неизвестный провайдер»
  - `clearKeyCache()` — для тестов и ручного обновления

- **`llmClient.js`** — порт `llm_client.py`. Унифицированный интерфейс на трёх провайдеров с одинаковым выходом `{text, inputTokens, outputTokens, cachedTokens}`:
  - `LLMError extends Error` — отдельный класс, чтобы caller (taskRunner в Сессии 26 / Сессии 3 roadmap'а команды) мог отличить «упал провайдер» от «упала наша логика»
  - `call({provider, model, systemPrompt, userPrompt, cacheableBlocks, maxTokens})` — главная функция, ветвится по провайдеру
  - **Anthropic prompt caching** реализован: `cacheableBlocks` (обычно `context.md` и `concept.md`) оборачиваются в `system: [{type:"text", text, cache_control: {type:"ephemeral"}}]`. SDK сам разрулит cache_read и в Usage отдаст `cache_read_input_tokens` / `cache_creation_input_tokens`. Считаем `total_input = input_tokens + cache_read + cache_creation` — Anthropic'овский `input_tokens` НЕ включает кешированные, и costTracker должен видеть полную картину для разбиения по ставкам
  - **OpenAI** — пробует сначала `max_completion_tokens` (новые модели), на `BadRequestError` про неподдерживаемый параметр фолбэчит на `max_tokens` (старые). Cached tokens забирает из `usage.prompt_tokens_details.cached_tokens` (если SDK прислал)
  - **Google** — `systemInstruction` идёт в `getGenerativeModel(...)`, не как сообщение. У Google AI Studio пока нет publicly доступного prompt caching через generate_content, поэтому `cacheableBlocks` для Google игнорируется; `cachedContentTokenCount` из `usageMetadata` всё равно читаем (на случай будущей поддержки)
  - **Обработка ошибок** — типизированная по классам SDK (`AuthenticationError`, `PermissionDeniedError`, `NotFoundError`, `RateLimitError`, `BadRequestError`, `APIConnectionError`), русские сообщения скопированы из Python-версии 1-в-1 (формулировки Влад уже видел в локальной ДК Лурье — оставляем привычными)
  - Ключи — через `keysService.getApiKey(provider)`, **не из env Railway**

- **`promptBuilder.js`** — порт `prompt_builder.py`:
  - `buildPrompt(templateName, variables)` → `{system, user, cacheableBlocks, template}`
  - Шаблоны читаются из bucket'а `team-prompts` через `teamStorage.downloadFile`. Имя без `.md` — добавляется автоматически
  - Регексп плейсхолдеров — `\{\{\s*([a-zA-Z_][a-zA-Z0-9_]*)\s*\}\}/g` (совпадает с Python-версией). Reset `lastIndex` перед каждым use — JS RegExp с флагом `g` сохраняет состояние между вызовами, забыть = плавающий баг
  - Разбиение шаблона `## System` / `## User` через case-insensitive multi-line regex. Если `## System` нет — весь файл считается system-телом, `userBody = null`. Если `## User` нет — user-сообщение это `{{user_input}}` (или пусто)
  - **Авто-загрузка `context.md` и `concept.md`** из team-database, если в шаблоне есть `{{context}}` / `{{concept}}` и эти переменные не переданы caller'ом. Файла нет (свежий проект, Влад ещё не написал) — пусто, не падаем
  - Cacheable блоки (`context`, `concept`) кладутся в `cacheableBlocks` массив. Пустые/whitespace-only не попадают (Anthropic ругается на whitespace-only system-блоки)
  - `listTemplates()` — список `.md` файлов в team-prompts, отсортирован

- **`costTracker.js`** — порт `cost_tracker.py`:
  - `loadPricing()` — читает `pricing.json` из bucket'а `team-config`, кеширует в памяти на 60 сек. Если файла нет / невалидный JSON — возвращает `{}` и логирует warning, стоимости будут 0
  - `calculateCost({provider, model, inputTokens, outputTokens, cachedTokens, audioMinutes})` — считает USD по схеме pricing.json `{models: [{provider, id, input_per_million, output_per_million, cached_input_per_million}], audio: {provider: {model: {per_minute}}}}`
  - **Биллинг кеша**: если есть `cached_input_per_million` (Anthropic) — `fresh = inputTokens - cachedTokens` оплачивается по обычному `input`, остальное по `cached_input`. Если нет (OpenAI/Google в pricing.json обычно не указывают) — все cached оплачиваются по полному `input` (как в Python-версии)
  - Округление до 6 знаков (доли цента)
  - `recordCall(...)` — считает стоимость + пишет в `team_api_calls` через `teamSupabase.recordApiCall`
  - `getCostForTask(taskId)` — сумма по всем вызовам с этим task_id. Используется для биллинга `write_text + AI-правки фрагментов`: правки биллятся к родительской задаче без новой записи в team_tasks
  - `getTotalSpending()` — агрегация по провайдерам и моделям + флаг `alert_triggered` (порог из team_settings.alert_threshold_usd). Структура ответа повторяет Python — фронт можно портировать 1-в-1
  - `getAlertThreshold()` — из team_settings, поддерживает форматы `{value: number}` (jsonb с обёрткой) и `number` (просто число)

- **`backend/scripts/testTeamLlm.js`** — тестовый CLI-скрипт:
  - Запуск: `node scripts/testTeamLlm.js [provider] [model]` (дефолт `anthropic claude-sonnet-4-5`)
  - Сначала вызывает `getAllKeysStatus()`, печатает статус всех трёх провайдеров (видно сразу, чего не хватает)
  - Если ключа нет — exit 1
  - Делает один вызов LLM с тестовым промптом «Скажи коротко: какой сегодня лучший способ начать день?»
  - Печатает ответ + токены (input/output/cached) + стоимость
  - Записывает вызов в `team_api_calls` через `recordCall`. taskId = null (это разовый тестовый, не задача из team_tasks)

Решения, принятые намеренно:
- **Singleton service-role клиента в `teamSupabase.js`**, не отдельный модуль клиента. У Потока уже есть `supabaseService.js` со своим клиентом (для videos), но мешать в один — связность хуже. Лучше повторить паттерн service-per-entity Потока, чем сэкономить пять строк
- **`appendTaskSnapshot` мапит camelCase JS → snake_case БД явным проходом**, не через автоматический преобразователь. Лишняя страховка от опечаток в полях `team_tasks` (их 17 — одна ошибка имени и поле уйдёт в null без явной ошибки)
- **getActiveTaskIds через клиентский DISTINCT ON.** Postgres позволяет `SELECT DISTINCT ON (id) ... ORDER BY id, created_at DESC`, но для этого нужен подзапрос с фильтром `WHERE status = 'running'` поверх — Supabase JS клиент не делает это удобно. Объёмы команды (десятки задач) — клиент-сайд группировка приемлема. Если станет узким местом — добавим Postgres-функцию
- **Ключи в БД, кеш на 30 сек.** Альтернатива — env Railway. Спека настаивает на БД (UI Админки + смена без передеплоя), и это согласовано в архитектурном документе. 30 сек — компромисс между «свежестью» и «не дёргать БД на каждом вызове»
- **`recordApiCall` не бросает.** Если запись в журнал упадёт (БД временно недоступна) — ломать LLM-вызов из-за лога нельзя. Клиент уже получил ответ модели, вернуть ему ошибку — потеря результата. Лог — журнал, а не критичная транзакция
- **Pricing-кеш 60 сек** (а не TTL=∞). Влад может обновить pricing.json через Supabase Dashboard в любой момент; чтобы свежие ставки подхватились без рестарта бэкенда — короткий TTL. На стоимости вызовов это не сказывается
- **Anthropic SDK errors через `instanceof`**, не через `name`-проверки. SDK экспортирует классы (`Anthropic.AuthenticationError` и т.д.) — `instanceof` устойчив к рефакторингам имён внутри SDK
- **OpenAI fallback `max_completion_tokens` → `max_tokens`** один уровень. У SDK нет публичного флага «какие параметры поддерживает модель», и я не хочу делать таблицу. Один try-catch покрывает оба пути
- **Google: `systemInstruction` через `getGenerativeModel`**, не через сообщения. Это правильный путь у Google AI Studio — system-инструкция не должна быть частью conversation history
- **Имя файла теста — `testTeamLlm.js`, не `test_team_llm.js`.** Стилистика Потока (`videoProcessor.js`, `apifyService.js`) — camelCase для скриптов и сервисов

Известные ограничения сессии 25:
- **Кеш ключей в памяти процесса** — на нескольких репликах Railway (если когда-нибудь будут) рассинхронизируется максимум на 30 сек. Для одного процесса не критично
- **Cleanup `team_tasks` отсутствует**. Журнал растёт линейно с числом задач — для команды это десятки в неделю, не проблема. Если когда-нибудь нужно будет архивировать — отдельная сессия с retention-политикой

- **`teamStorage.downloadFile` возвращает строку**. Бинарные файлы (например, PDF в `team-database/sources/`) пока через этот метод нормально не вытащить — для них в Сессии 26 / 27 (Сессии 3 / 4 roadmap'а команды) добавим `downloadBlob` или `downloadBuffer`. **Сессия 26: Команда — TaskRunner и handlers** (завершена 2026-05-05)

**Эта сессия — Сессия 3 из 8 в roadmap-е этапа 1 раздела «Команда»**, в нумерации сессий Потока — Сессия 26. Базируется на сервисах из Сессии 25 (LLM-клиент, prompt builder, cost tracker, teamSupabase, teamStorage, keysService) и инфраструктуре Сессии 24 (таблицы `team_*` и buckets `team-*`).

Эта сессия — оркестратор задач команды: TaskRunner с приёмами «append-only журнал состояний» и «изолированные статусы шагов», пять handler'ов под каждый тип задачи, очередь и recovery после рестарта. Без HTTP-роутов пока — они будут в Сессии 27 (Сессия 4 roadmap'а команды). Запуск задач сейчас идёт только из тестового скрипта `backend/scripts/testTeamTask.js`.

- **Новая зависимость в `backend/`**: `pdf-parse` (^1.1.4) — для извлечения текста из PDF в `contentFetcher.js`. Бинарные превью/ассеты этим пакетом не вытаскиваются — он умеет только текст. Альтернатива (`pdf2json`) тяжелее и медленнее; pdf-parse даёт чистый text out-of-the-box

- **`backend/src/services/team/contentFetcher.js`** — порт `content_fetcher.py` на JS:
  - `fetchSource(source)` — единая точка входа. Если строка — URL (http/https) → `fetchUrl`, иначе пробуем как путь в bucket'е `team-database` (с авто-префиксом `sources/` если не указан).
  - `fetchUrl(url)` через встроенный `fetch` (Node 20+) с браузерным User-Agent, таймаутом 30 сек через AbortController, `redirect: "follow"`. По content-type или расширению .pdf — переключается на pdf-парсер (через ленивый `import("pdf-parse")`), иначе вытаскивает текст HTML простыми регекспами
  - **HTML-парсер регексповый, не JSDOM**. JSDOM весит ~50 МБ и медленный, для нашей задачи (вытащить читаемый текст) перебор. Регексп-подход покрывает 95% реальных страниц: удаляем шумные блоки (`<script>`, `<style>`, `<nav>`, `<footer>`, `<aside>`, и т.д.), сужаемся до `<article>` или `<main>` если найдены, превращаем блочные тэги в переводы строк, декодируем HTML-entities. Title из `<title>` идёт в шапку текста с `=`-разделителем. Свёртка пустых строк (3+ → 2)
  - **PDF из storage** требует бинарные байты, а `teamStorage.downloadFile` отдаёт UTF-8 строку (через `Blob.text()`). Поэтому в `contentFetcher` свой helper `downloadFileAsBuffer(path)` — обращается напрямую к service-role клиенту через `getServiceRoleClient` и берёт `arrayBuffer`. `teamStorage.downloadFile` не трогаю — у него своё назначение (текстовые файлы), и менять его сигнатуру = менять везде, где он используется
  - **MAX_TEXT_CHARS = 200_000** — обрезание длинного текста (≈50 тыс токенов), чтобы не разорвать промпт. Совпадает с Python-версией. Сообщение об обрезании добавляется в конец как `…[обрезано на 200000 символах из 587342]`
  - Класс `FetchError extends Error` — отдельный тип, чтобы taskRunner мог отличить «упал источник» от «упала наша логика»

- **`backend/src/services/team/taskHandlers.js`** — пять handler-функций + реестр + хелперы:
  - `TASK_HANDLERS` реестр: `ideas_free`, `ideas_questions_for_research`, `research_direct`, `write_text`, `edit_text_fragments`. **Точная копия логики Python**: те же имена ключей, та же сигнатура `async (task) → {result, artifactPath, tokens, prompt?}`. Это критическая точка расширения для этапа 2 (новые типы под AI-агентов добавляются регистрацией в реестре + новый шаблон в `taskTemplateName`)
  - `TASK_TITLES` — человекочитаемые названия для UI (используется как дефолт `task.title` если пользователь не задал свой)
  - `taskTemplateName(taskType)` — маппинг типа на имя файла шаблона в bucket'е `team-prompts`
  - `HIDDEN_TYPES_IN_LOG` — `Set(["edit_text_fragments"])`. AI-правки фрагментов привязаны к родительской write_text задаче и не должны показываться в канбане как самостоятельные карточки. На этапе 2 при добавлении технических типов под агентов — пополнить
  - **Артефакты идут в bucket `team-database`**, а не на локальный диск (как было в ДК Лурье). `artifactPathFor(taskType, params)` строит путь относительно корня bucket'а: `ideas/<ts>_<slug>.md`, `research/<ts>_<slug>.md`, `texts/<slug>/v<N>_<ts>.md`. Этот путь и сохраняется в `team_tasks.artifact_path` — без bucket-префикса (он подразумевается)
  - `slugify(text, 40)` — Unicode-aware: `[^\p{L}\p{N}\s_-]/gu` сохраняет кириллицу. Без флага `u` русские названия точек экскурсий превращались бы в "untitled-point" целиком — критично для UX
  - `nextVersion(pointDir)` — для версионирования write_text. Через `teamStorage.listFiles(bucket, pointDir)` (Supabase API → `storage.list`) получаем все файлы папки, ищем максимальный `v<N>_` префикс, +1. Если папки нет (первая версия) — возвращаем 1
  - **Особенность `edit_text_fragments`**: handler ожидает `params.parent_artifact_path` (полный путь до родительского артефакта в bucket'е). Сам он в БД не лезет — оркестратор `taskRunner.runTaskInBackground` подкладывает это значение в params **перед вызовом** handler'а после lookup'а родителя. Так handler остаётся «тупым» и тестируемым в изоляции
  - `gatherResearch(paths)` — собирает блок исследований для write_text: проходит по списку путей в `team-database`, скачивает каждый файл через `downloadFile`, склеивает с разделителями `### Источник: <title>`. Ошибки и отсутствующие файлы пропускаются молча (как в Python — `try/except: continue`)
  - `formatEdits(edits)` — порт Python `_format_edits`: рендерит структурированный список правок в numbered markdown-блок для шаблона. Экспортируется отдельно — нужен и в handler, и в taskRunner.applyFragmentEditsInline, и в previewPrompt
  - `buildPreviewVariables(taskType, params)` — единый помощник для `previewPrompt` и handler'ов: автоматически разворачивает `research_paths` в `research`-блок для write_text и форматирует `edits` для edit_text_fragments. Это позволяет UI показать ровно тот промпт, который улетит в LLM

- **`backend/src/services/team/taskRunner.js`** — оркестратор:
  - `createTask({taskType, params, modelChoice, promptOverride, title})` — резолвит модель через `resolveModelChoice`, собирает или берёт из override промпт, генерирует id (`tsk_<random>` через `crypto.randomBytes(6)`), пишет первый `running` снапшот в `team_tasks` и кладёт id в `teamWorkerPool`. Возвращает task id (UI-эндпоинт ответит 202 с этим id)
  - `runTaskInBackground(taskId)` — главный воркер задачи (вызывается из пула):
    - Читает текущий снапшот из БД через `getTaskById`
    - Для `edit_text_fragments` достаёт parent через `getTaskById(parent_task_id)`, валидирует тип/наличие артефакта и подкладывает `parent_artifact_path` в params текущей задачи (не сохраняя — чисто для handler'а)
    - Дёргает handler из `TASK_HANDLERS`
    - Записывает API-вызов в `team_api_calls` через `costTracker.recordCall` — это даёт `cost_usd`, который попадёт в сам снапшот `team_tasks`
    - Пишет `done`-снапшот через `appendUpdate` (helper, который читает текущий снапшот и мерджит обновления — это сохраняет «append-only» паттерн)
    - При ошибке — `recordCall(success: false, error)` + `appendUpdate(status: "error", error)`. **Не бросает наружу** — рекавери очереди/пул не должны падать из-за прикладной ошибки
  - `appendUpdate(taskId, update)` — внутренний хелпер-апдейтер: каждый переход состояния (started, done, error, archive, rename, mark_done, refresh_cost) пишет новую строку с тем же `id`. Текущее состояние = последний снапшот (читается через `getTaskById` → `ORDER BY created_at DESC LIMIT 1`). Это и есть append-only журнал
  - `mergeSnapshot(current, update)` — мапит snake_case БД → camelCase для `appendTaskSnapshot`. Поля, которые `update` не указывает, наследуются из текущего снапшота
  - `archiveTask(taskId)`, `renameTask(taskId, title)`, `markTaskDone(taskId)` — каждая просто обёртка над `appendUpdate` со своим набором полей
  - `refreshTaskCost(taskId)` — пересчитывает суммарный `cost_usd` задачи как `getCostForTask(taskId)` (сумма по всем `team_api_calls` с этим task_id) и пишет новый снапшот. Используется после AI-правок фрагментов
  - **`applyFragmentEditsInline({parentTaskId, fullText, edits, generalInstruction, modelChoice, promptOverride})`** — особый путь: применяет AI-правки к артефакту write_text задачи. **Не создаёт новую запись в `team_tasks`** — только новую версию файла в той же папке точки + запись в `team_api_calls` с `task_id = parentTaskId`. После этого вызывается `refreshTaskCost(parentTaskId)`, и UI карточки родительской задачи показывает суммарную стоимость (write_text + все правки)
  - **`saveDirectEdit({parentTaskId, content})`** — прямая правка от пользователя без LLM-вызова. Просто новая версия `vN+1_<ts>.md` в папке точки, биллинг = 0
  - **`appendQuestionToResearch({parentTaskId, question, modelChoice})`** — дополнительный вопрос к research_direct задаче. Не создаёт запись в `team_tasks` — дописывает блок «## Вопрос (HH:MM, DD.MM)» в существующий артефакт + запись в `team_api_calls` к parent task id + `refreshTaskCost(parentTaskId)`. Источник (URL/файл) переиспользуется из `parent.params.source` — `fetchSource` качает повторно. Если хочется без перекачки источника — пользователь редактирует промпт перед запуском (overrideUsed)
  - **`resolveModelChoice(modelChoice, taskType)`** — порт Python-функции:
    - Кеш `presets.json` и `pricing.json` в памяти на 60 сек
    - Поддерживает три формы: `{preset}`, `{model}` (провайдер ищется в pricing.json), `{provider, model}` (явный)
    - Per-task override в presets: `presets[name][taskType]` приоритетнее `presets[name].default`
    - Legacy nested form `{models: {provider: id}}` поддерживается для обратной совместимости со старыми presets.json
  - **`previewPrompt(taskType, params)`** — собрать промпт без запуска задачи. Используется UI'ным эндпоинтом превью. Внутри: `taskTemplateName` → `buildPreviewVariables` → `buildPrompt`. Это даёт пользователю возможность увидеть и отредактировать ровно тот текст, который улетит в LLM

- **`backend/src/queue/teamWorkerPool.js`** — отдельный пул задач команды:
  - Структура копирует `workerPool.js` Потока 1-в-1 (queue + dedup Set + activeWorkers + concurrency + processFn)
  - **Отдельный пул, не расширение существующего**. Видео-обработка (Apify+Whisper+OpenAI saммари) и команда (Anthropic/OpenAI write_text) не должны блокировать друг друга. Если оба разделят один пул с concurrency=2 — батч из 100 видео может встать в очередь перед задачей пользователя «напиши текст», и UI зависнет в `running` на часы
  - **Дефолт concurrency=1** (через `TEAM_WORKER_CONCURRENCY`) — задачи команды длиннее (write_text идёт 30–60 сек) и дороже (один write_text ≈ $0.01–0.05 vs $0.001 за видео), параллельный запуск быстро упрётся в rate-limit Anthropic и неприятно ударит по биллингу

- **`backend/src/services/team/teamRecoveryService.js`** — recovery после рестарта:
  - `recoverUnfinishedTeamTasks()` — при старте бэкенда сканирует `team_tasks` через `getActiveTaskIds` (там работает client-side DISTINCT ON: для каждого id берётся последний снапшот, отбираются те, у кого status = `running`). Эти id заливаются в `teamWorkerPool` через `enqueueTeamTasks`
  - Логика 1-в-1 с `recoveryService.js` Потока. Без recovery после рестарта Railway зависшие в `running` задачи остались бы там навсегда — состояние очереди только в памяти
  - Ошибки логируются, но старт сервера не валят (recovery — не критичная функция)

- **`backend/src/index.js`** обновлён: после конфигурации video-пула добавлены `configureTeamWorkerPool` (с `runTaskInBackground` как processFn и `env.teamWorkerConcurrency`) и `recoverUnfinishedTeamTasks()` после `app.listen`. В стартовых логах появилась строка `Team pool : concurrency=1`

- **`backend/src/config/env.js`** обновлён: добавлено `teamWorkerConcurrency: parseInt(env.TEAM_WORKER_CONCURRENCY ?? "1")`. В `.env.example` появилась переменная с подробным комментарием про дефолт=1 и когда поднимать

- **`backend/scripts/testTeamTask.js`** — тестовый CLI-скрипт:
  - Конфигурирует teamWorkerPool с runTaskInBackground как processFn (без HTTP-сервера — нам нужна только сама очередь)
  - Создаёт задачу `ideas_free` с тестовым вводом «Придумай 3 идеи коротких видео-сюжетов про необычные истории Санкт-Петербурга XIX века» и пресетом `balanced` (по умолчанию; можно переопределить через `node scripts/testTeamTask.js fast`)
  - Поллит статус каждые 2 сек через `getTaskById`, печатает изменения статуса (`running` → `done`)
  - Финальный вывод: статус, provider/model, токены, стоимость, путь до артефакта, первые 500 символов ответа
  - Безопасно повторно запускать — каждый запуск это новая запись в `team_tasks` (две: running + done), артефакт в `team-database/ideas/`. Удалить можно через Dashboard

Решения, принятые намеренно:
- **TASK_HANDLERS реестр — точная копия Python.** Имена ключей (`ideas_free`, `ideas_questions_for_research`, `research_direct`, `write_text`, `edit_text_fragments`), сигнатура async-функции, формат outcome `{result, artifactPath, tokens, prompt?}` — всё совпадает. Это позволяет в этапе 2 портировать UI-флоу команды агентов, не переписывая ядро. Если поменять имена ключей — миграции за миграциями
- **Append-only журнал, не in-place update.** Альтернатива — UPDATE одной строки в `team_tasks` при каждом переходе. Но историю состояний хорошо иметь: видеть, как задача шла (running → error vs running → done → marked_done → archived). На этапе 2 пригодится для ретроспективы агентов и для memory. Хранилище дешёвое — десятки строк на задачу не разорят бюджет
- **Изолированные статусы шагов закладываются сейчас**, даже если задачи однофазные. Каждый снапшот хранит `started_at`, `finished_at`, `error` — это поля задачи, не шага, но архитектурно paving the way: на этапе 2 при многошаговых задачах добавится `step_status` или вынесется в отдельную таблицу `team_task_steps`. Пока избыточная инфраструктура не плодится
- **Отдельный teamWorkerPool, не расширение existing'а.** См. выше. Кратко: видео-обработка и команда — независимые pipeline'ы с разными rate-limit'ами, разной конкурентностью и разными SLA. Один пул на двоих будет блокировать друг друга
- **edit_text_fragments handler «тупой» — parent artifact пробрасывается через params оркестратором.** Альтернатива — handler сам делает getTaskById parent'а. Но handler должен быть чистой функцией от task → outcome, чтобы его можно было вызвать в разных контекстах (live-задача, replay, тестирование). Подкладывание parent_artifact_path в params оркестратором перед вызовом сохраняет это свойство
- **`applyFragmentEditsInline` и `appendQuestionToResearch` не создают новый снапшот в `team_tasks`.** Биллинг идёт против `parentTaskId` через `team_api_calls`, и `refreshTaskCost` обновляет суммарный cost в parent'е. Альтернатива — отдельные карточки в канбане, что захламляет UI. AI-правки и follow-up вопросы — это интеракции в рамках одной задачи, не отдельные сущности
- **HTML-парсер регексповый, не JSDOM/cheerio.** JSDOM ~50 МБ + медленный startup, cheerio лёгкий, но всё равно лишняя зависимость. 95% полезных страниц с историческим/культурным контентом (Wikipedia, статьи) хорошо парсятся регекспами. Если на каком-то источнике результат будет плохой — добавим cheerio точечно
- **pdf-parse через ленивый `import()` внутри `extractPdfBytes`.** Если по каким-то причинам пакет не установлен или сломан, остальной функционал команды не упадёт — упадёт только PDF-источник. Раньше ловить такое = диагностика на проде
- **Test-script конфигурирует пул сам, без поднятия Express.** Запуск scripts/testTeamTask.js не должен поднимать HTTP-сервер. Альтернатива — выносить `configureTeamWorkerPool` в `index.js` и заставлять тест запускать `npm start` параллельно — лишнее
- **`crypto.randomBytes(6).toString("hex")` для task id.** 12 hex-символов = 48 бит энтропии. Для общего числа задач команды (тысячи в течение долгого времени) коллизия маловероятна. Если когда-нибудь понадобится больше — увеличить до 8 байт. UUID был бы избыточным

Известные ограничения сессии 26:
- **Если бэкенд перезапустится посреди обработки задачи команды — текущая LLM-вызов теряется**, статус снапшота останется `running`. После рестарта `teamRecoveryService` подхватит и **запустит её с нуля** (промпт пересоберётся, артефакт перезапишется в Storage с upsert: true). Чекпойнтов между шагами нет — для текущего масштаба перебор. Биллинг успевшего LLM-вызова всё равно записан в `team_api_calls` (если cost_tracker.recordCall успел) — деньги «потеряны»
- **`appendQuestionToResearch` всегда заново качает источник.** Альтернатива — сохранять fetched-текст в `parent.params.source_text` после первого fetch'а. Но `params` могут быть большими (URL → 200 КБ текста), и хранить их в БД ради экономии второго fetch'а спорно. На 95% случаев пользователь задаёт follow-up через минуту — re-fetch упирается в кеш браузера/CDN сервера и быстрый
- **HTML-парсер не понимает SPA.** Сайты, отдающие пустую `<body>` и наполняющие её через JS, дадут пустой текст. Это редкость для исторического/культурного контента (там обычно SSR), но если источник окажется SPA — пользователь увидит «(пусто)» и поймёт, что нужен другой источник
- **Реальный фронт-эндпоинт для запуска задач ещё не подключен.** Запуск из браузера/UI заработает в Сессии 27 (Сессия 4 roadmap'а команды). Сейчас единственный способ запустить задачу — `node scripts/testTeamTask.js`

**Сессия 27: Команда — REST API маршруты** (завершена 2026-05-05)

**Эта сессия — Сессия 4 из 8 в roadmap-е этапа 1 раздела «Команда»**, в нумерации сессий Потока — Сессия 27. Базируется на сервисах из Сессий 25–26 (TaskRunner, handlers, teamSupabase, teamStorage, costTracker, keysService) и инфраструктуре Сессии 24 (таблицы `team_*` и buckets `team-*`).

Эта сессия — связывание бэкенд-ядра команды с внешним миром через REST-эндпоинты Express. Фронт ещё не написан (Сессии 28-31), но после этой сессии все операции команды можно дёргать через curl/Postman, и сторонним инструментам (если когда-нибудь понадобятся) тоже доступен полный API.

- **Новая зависимость в `backend/`**: `multer` (^2.1.1) для multipart-загрузки файлов и аудио. Альтернативу `formidable` рассматривал — multer проще интегрируется со стилем существующих роутов Потока (паттерн `upload.single("file")` через middleware).
- Адаптация `backend/src/services/transcriptionService.js` — добавлена функция `transcribeFromBuffer(buffer, originalName)`. Существующая `transcribeFromUrl` оставлена без изменений (нужна для Whisper в видео-обработке Потока). Новая версия:
  - Принимает буфер с аудио (webm/ogg/mp3/wav/m4a) от MediaRecorder API из браузера или multipart upload'а
  - Пишет буфер во временный файл (расширение определяется по `originalName`, либо `.bin` если неизвестно — ffmpeg тогда определит формат по содержимому)
  - Прогоняет через тот же `extractAudio` (mp3 64 kbps моно 16 kHz) — переиспользуем существующий ffmpeg-pipeline
  - Дополнительно через `getMediaDurationSeconds` парсит stderr ffmpeg (`Duration: 00:01:23.45`) — нужно для биллинга Whisper по минутам, costTracker умножит на ставку из pricing.json
  - Возвращает `{text, durationSeconds}`. **Не возвращает no_speech-объект** как сестринская URL-функция — для голосовой заметки команда хочет либо текст, либо явную ошибку, а не молчаливое «нет речи»
- **Шесть новых роутов** в `backend/src/routes/team/`:
  - **`tasks.js`** (8 эндпоинтов):
    - `GET /api/team/tasks/templates` — список доступных типов задач (читает `TASK_HANDLERS` + `TASK_TITLES` + `HIDDEN_TYPES_IN_LOG`). UI рендерит кнопки «Идеи (свободные)» / «Исследовать напрямую» / etc по этому списку
    - `POST /api/team/tasks/preview-prompt` — `{taskType, params}` → собранный промпт без запуска. Дёргает `taskRunner.previewPrompt`
    - `POST /api/team/tasks/run` — `{taskType, params, modelChoice?, promptOverride?, title?}` → `{taskId}`. Создаёт задачу через `createTask`, пишет первый running-снапшот, кладёт id в очередь команды, возвращает 202
    - `POST /api/team/tasks/:taskId/archive` — `{task}` (новый снапшот status=archived)
    - `POST /api/team/tasks/:taskId/rename` — body `{title}`
    - `POST /api/team/tasks/:taskId/mark-done` — для статуса `marked_done`
    - `POST /api/team/tasks/:taskId/append-question` — body `{question, modelChoice?}`. Дёргает `appendQuestionToResearch` — дописывает в существующий артефакт research_direct, биллинг к родителю, без новой записи в team_tasks
    - `POST /api/team/tasks/:taskId/apply-ai-edit` — body `{fullText, edits, generalInstruction?, modelChoice?, promptOverride?}`. Создаёт новую версию артефакта write_text, биллинг к родителю
    - `POST /api/team/tasks/:taskId/save-direct-edit` — body `{content}`. Прямая правка без LLM, новая версия в той же папке точки, биллинг = 0
    - **Валидация taskId через regex** `/^tsk_[A-Za-z0-9]+$/` — task id формата генерируется в `taskRunner.newTaskId()`, регэксп его покрывает. Защищает от path-injection в URL
  - **`artifacts.js`** (5 эндпоинтов):
    - `POST /api/team/artifacts` — `{path, content}` → upload в `team-database`
    - `DELETE /api/team/artifacts` — `{path}` → удаление файла
    - `POST /api/team/artifacts/move` — `{fromPath, toPath}` → download → upload to new → delete old. Дополнительно обновляет `artifact_path` во всех снапшотах `team_tasks`, ссылавшихся на старый путь — иначе UI старых задач не найдёт свой файл. Если `team_tasks` обновить не получилось (RLS, рассинхрон) — логируем warning, файл всё равно перенесён
    - `POST /api/team/artifacts/folders` — `{path}` → создаёт «папку» через upload плейсхолдера `<path>/.keep` (Storage не имеет явных папок). Если плейсхолдер уже есть — возвращает `alreadyExists: true`
    - `DELETE /api/team/artifacts/folders` — `{path}` → удаление **пустой** папки. Сначала через `listFiles` проверяем, что внутри ничего нет кроме `.keep`; если есть — 409 «Папка не пуста». Безопасный паттерн: пользователь не может одной кнопкой смести важные данные
    - **Sanitize path-input** — функция `sanitizePath` отбрасывает пустые строки, ведущие слэши, `../` (path traversal), пробелы. Возвращает нормализованный путь или null
  - **`prompts.js`** (1 эндпоинт):
    - `POST /api/team/prompts` — `{name, content}` → upload в `team-prompts`. Имя проверяется regex `/^[A-Za-z0-9_-]+(?:\.md)?$/` (плоская структура шаблонов, никаких слэшей). Если расширения нет — добавляем `.md`. Чтение списка и контента шаблонов фронт делает напрямую через teamPromptsService — здесь только запись
  - **`admin.js`** (6 эндпоинтов):
    - `GET /api/team/admin/keys` — статус с маскированным preview (`{anthropic: {configured: bool, masked: "sk-anth***...***xyzv", updatedAt}, openai: {...}, google: {...}}`). Маскирование: первые 4 + `***` + последние 4 символа (для коротких ключей <8 знаков — `xx***`). Сами ключи никогда не отдаём из API
    - `POST /api/team/admin/keys` — `{provider, key}`. Делегирует `keysService.setApiKey`, тот сбрасывает кеш на 30 сек. Поддерживаемые провайдеры — `anthropic | openai | google`
    - `DELETE /api/team/admin/keys` — `{provider}`
    - `GET /api/team/admin/keys-status` — компактная версия `{anthropic: bool, openai: bool, google: bool}` для индикатора в шапке
    - `GET /api/team/admin/spending` — `costTracker.getTotalSpending()` (total/by_provider/by_model + alert_triggered)
    - `GET /api/team/admin/alert-threshold` → `{value: number | null}` из team_settings
    - `POST /api/team/admin/alert-threshold` — `{value: number | null}`. Валидация: положительное число или null (выключить алерт)
  - **`files.js`** (1 эндпоинт через multer middleware):
    - `POST /api/team/files/upload` — multipart/form-data: поле `file` (обязательно), поле `prefix` (опц., дефолт `uploads/`). Имя файла санитизируется через `sanitizeFilename` (basename без директорий, без NUL-байтов; кириллица сохраняется). Дописывается timestamp перед расширением (`файл-1714939200000.pdf`) — защита от коллизий имён при повторной загрузке
    - **Лимиты multer**: 50 МБ на файл (PDF интервью, сканы — типичный размер), 1 файл за запрос. memoryStorage — буфер сразу в Storage без записи на диск
    - Перехватчик ошибок multer возвращает понятный JSON `{error: "Файл слишком большой..."}` со статусом 413, иначе пользователь получил бы HTML-страницу Express
  - **`voice.js`** (1 эндпоинт через multer middleware):
    - `POST /api/team/voice/transcribe` — multipart/form-data: поле `audio`. Лимит 25 МБ (= лимит Whisper). Дёргает `transcribeFromBuffer`, пишет вызов в `team_api_calls` через `costTracker.recordCall` (provider=`openai`, model из `env.whisperModel`, audioMinutes из ffmpeg-stderr). Возвращает `{text, durationSeconds, costUsd}`
    - На неудачу — пишет `recordCall(success: false, error)` для статистики падений Whisper, потом 500 с понятным сообщением

- **Подключение в `backend/src/app.js`**: импорты + `app.use("/api/team/...", router)` для всех шести роутеров. Также увеличен лимит JSON-парсера с 1 МБ до **5 МБ** — write_text промпты с research-блоками (большие транскрипции + контекст блога + полный текст для редактирования) могут перешагивать 1 МБ. 5 МБ — щедрый запас, на JSON-API без файлов не давит, файлы и аудио идут через multer-stream и не зависят от json-parser-лимита

Решения, принятые намеренно:
- **Шесть отдельных файлов вместо одного `team.js`.** Каждый файл — одна логическая зона ответственности (задачи / артефакты / шаблоны / админка / файлы / голос). Файлы по 100-250 строк удобнее для нахождения нужного эндпоинта и для будущих правок (в Сессии 28+ фронт-сервисы будут добавлять новые маршруты — лучше точечно расширять файл, чем туда-сюда лазать в 1000-строчном модуле)
- **Чтение через прямой Supabase, запись через бэкенд.** Прямой паттерн Потока: фронт читает `videos` из browser-supabase с RLS, а пишет через server actions / бэкенд. Списки задач, файлы артефактов, шаблоны промптов — всё чтение идёт мимо бэкенда. На бэкенде только тяжёлые операции, требующие service-role: запуск задач (запись team_tasks), ключи (RLS закрыта по соображениям безопасности), запуск Whisper, fetch URL/PDF
- **Multer вместо formidable.** Multer проще интегрируется через middleware-паттерн `upload.single("file")` — это вписывается в стиль существующих Express-роутов. Formidable требует нетривиальной обработки потока вручную. На сравнимой функциональности multer выигрывает чистотой кода
- **`transcribeFromBuffer` отдельная функция, не общая.** Альтернатива — обобщить `transcribeFromUrl` через перегрузку. Но это разные сценарии: для URL нужен Instagram CDN-санитайзинг + no_speech-эвристика (видео без речи бывает); для буфера — durationSeconds для биллинга. Совмещать = усложнять оба пути. Лучше две тонкие функции с общим helper'ом (`extractAudio`)
- **Все эндпоинты под `/api/team/*`.** Чёткая граница с существующими `/api/videos`, `/api/thumbnails`, `/api/reprocess-references`. Также упрощает мониторинг и потенциальный rate-limiting: в будущем можно навесить middleware только на `/api/team/*` (например, общий auth-middleware), не трогая остальные пути
- **Path traversal-защита через явный sanitize.** Альтернатива — доверять Supabase Storage в плане валидации путей. Но нет смысла полагаться на чужой клиент — лишний слой проверок дёшев и снимает целый класс атак
- **JSON-лимит 5 МБ.** Альтернатива — оставить 1 МБ и заставить фронт резать write_text-вызовы. Но 5 МБ это всё равно меньше доли секунды на парсинг, и на JSON-API без больших файлов не давит. Разумная инвестиция в спокойствие
- **`task_id`-validation regex `/^tsk_[A-Za-z0-9]+$/`.** Защита от path-injection через `:taskId` в URL. Если пользователь подсунет `../../../tasks/something` — regex отвергнет, не дав даже коснуться taskRunner. Возможность уязвимости минимальная (taskRunner внутри сам проверяет, существует ли задача), но дополнительная страховка дешевая

Известные ограничения сессии 27:
- **Нет аутентификации эндпоинтов**. RLS открыта на anon/authenticated в team_* таблицах (как и для videos/bookmarks), и бэкенд тоже открыт. Это согласовано на этапе модели «личный инструмент без auth» — Поток с самого начала так живёт. Если когда-нибудь понадобится auth — будем навешивать middleware на `/api/team/*`
- **Загрузка файлов и аудио идут в memory.** При больших объёмах (≥50 МБ) бэкенд будет потреблять память пропорционально. На Railway free tier это потенциально упрётся в лимит памяти. Решение «когда заболит» — переключиться на disk-storage multer'а или streamingmode
- **Move артефакта не атомарен**. Если между upload в новый путь и delete старого что-то упадёт (rare, но возможно) — останется два файла. Лечится повторным move или ручным удалением через Supabase Dashboard. Транзакций между Storage-операциями Supabase не предоставляет
- **Нет endpoint'а для bulk-операций**. Удаление пачкой, перемещение пачкой — пока не добавил, потому что фронт ещё не написан. Если в Сессии 30 (Сессия 7 roadmap'а команды) UI потребует — допилим точечно
- **Folder-плейсхолдер `.keep`**. Если пользователь зайдёт в Supabase Dashboard и создаст файл напрямую без `.keep` — папка появится в списке, но «удалить пустую папку» не сможет (плейсхолдера нет, удалять нечего). Это edge case, возникнет только при ручных правках через Dashboard

Полные обвязки (run, archive, applyAiEdit, voiceTranscribe, …) появятся в Сессиях 6–7 — здесь экспортируется только готовая инфраструктура `backendFetch()` с обработкой ошибок и таймаутом 15 с, поверх неё caller добавит конкретные методы

- **`frontend/src/lib/team/format.ts`** — `formatUsd(value)` (адаптивные знаки после запятой: <$0.01 — 4 знака, <$1000 — 2 знака, выше — без копеек) и `pluralize(count, [один, два-четыре, ноль/пять-десять])` — русское склонение для шаблонов «5 задач», «1 задача» и т. п.

- **`frontend/src/components/blog/team/`** — каркасные компоненты:
  - `TeamPageHeader.tsx` — унифицированная шапка страниц Команды (заголовок, breadcrumb, описание, опциональные actions). Совпадает по сетке и отступам с шапкой `/blog/references`. Аргумент `showBackLink` рисует ссылку «← Команда» вместо breadcrumb «Блог · Команда» — для внутренних страниц
  - `TeamSectionCard.tsx` — карточка-ссылка на внутренний раздел: иконка в акцентной плашке, название, описание, опциональный badge снизу. Hover-эффект (подъём на 1px, рамка темнее). При `highlight=true` рамка акцентная — используется на главной для «Админки», когда не настроены ключи
  - `TeamStatTile.tsx` — статистическая плашка (иконка + значение + label + hint). Используется на главной в ряду из трёх метрик
  - `TeamComingSoon.tsx` — заглушка для внутренних страниц на этапе каркаса. Иконка, заголовок, плашка «Что здесь будет» с буллетами, ссылка на номер сессии в roadmap'е. В Сессиях 6–7 заглушка заменяется на полноценный workspace, переиспользуемые шапка и сетка остаются

- **`frontend/src/app/blog/team/page.tsx`** — главная Команды (server component, `dynamic = "force-dynamic"`):
  - Шапка через `TeamPageHeader` без backlink
  - Ряд из трёх `TeamStatTile`: «В работе» (количество запущенных задач + хинт «всего N задач»), «Расходы» ($ за 30 дней), «Ключи» («Все на месте» / «Не хватает» / «—» если бэкенд недоступен). Иконка «Ключи» переключается на `AlertTriangle`, если хоть один ключ отсутствует
  - Сетка 2×2 из `TeamSectionCard`: Задачи / Промпты / База / Админка. Карточка «Задачи» показывает badge с количеством активных, если есть; карточка «Админка» подсвечивается акцентным бордером и badge «Нужно настроить ключи», если хоть один провайдер не настроен
  - Параллельно через `Promise.all` дёргает `fetchTeamOverview()` (Supabase) и `fetchKeysStatusSafe()` (бэкенд). Если бэкенд упал — страница продолжает работать, в карточке ключей показывается «—»

- **Страницы-заглушки** `/blog/team/{tasks,prompts,database,admin}` — server components, минимальный контент: `TeamPageHeader` со ссылкой назад + `TeamComingSoon` со списком запланированных фич. Каждая помечена `dynamic = "force-dynamic"` для консистентности с главной (статика тут не нужна, они будут дёргать данные в следующих сессиях)

Решения, принятые намеренно:
- **Главная Команды (`/blog/team`) — не пункт меню.** Меню показывает четыре подраздела напрямую — это убирает один лишний клик в типичном сценарии «Влад зашёл за задачей → открыл Канбан». Возврат на главную возможен через breadcrumb «← Команда» в шапке любой подстраницы. Альтернативный вариант — добавить «Главная» отдельным пунктом — был бы избыточен для четырёх-разделового меню
- **Чтение задач и расходов — напрямую из Supabase, не через бэкенд.** Так уже устроены `videosService.ts` и `manualFieldsService.ts` Потока — RLS открыта, anon-ключ безопасен, отклик мгновенный, лишнего хопа нет. Бэкенд-клиент дергаем только для service-role-операций (запуск задач, ключи, запись файлов)
- **Dedupe `team_tasks` на клиенте.** Postgres мог бы вернуть `DISTINCT ON (id) ... ORDER BY id, created_at DESC` за один проход, но Supabase JS клиент это не покрывает напрямую (нужен RPC). На десятках задач разница незаметна, переусложнять не стал — повторяет паттерн backend-сервиса `getActiveTaskIds()`
- **Statусы ключей через бэкенд (`fetchKeysStatusSafe`), не через Supabase.** Хотя `team_api_keys` тоже доступна через RLS, читать ключи из браузера — потенциальная утечка (даже если RLS открыта, в HTML могут попасть значения). Бэкенд возвращает только `{anthropic: bool, openai: bool, google: bool}` без самих значений. Дополнительный плюс — `fetchKeysStatusSafe()` ловит ошибки и страница рендерится даже если Railway временно недоступен
- **Каждая страница `dynamic = "force-dynamic"`.** Команда — живой инструмент с агрегатами и счётчиками; статический prerender дал бы устаревшие цифры. На Vercel это означает, что страница рендерится при каждом запросе и не кешируется. Производительность окей: один запрос в Supabase + один в бэкенд (для главной), под 200мс
- **Подразделы — заглушки, не пустые маршруты.** Альтернатива: оставить только главную, ссылки вели бы в 404. Но Влад тестирует UI и хочет видеть, что произойдёт после клика. Заглушки с понятным «Что здесь будет» дают ему карту разработки и ссылка с главной никогда не приведёт в никуда
- **Отдельная папка `lib/team/`** для сервисов и типов Команды. У Потока в `lib/` лежат videosService, manualFieldsService, types и прочее без вложенности — это работало для одного раздела. Для Команды появится 5–10 сервисов плюс типы — без папки `lib/` превратится в свалку. Папка `components/blog/team/` повторяет паттерн `components/blog/analyst/`

Известные ограничения сессии 28:
- **Подсчёт задач за полный список.** `fetchTeamTasks()` тянет все строки `team_tasks` (включая исторические снапшоты) и группирует. Если журнал когда-нибудь распухнет до тысяч строк — добавим Postgres-функцию `team_tasks_current()` или серверную view. Сейчас десятки задач × 5–10 снапшотов = сотни строк, нагрузка незаметна
- **Расходы — фиксированное окно «последние 30 дней».** Главная не позволяет переключить период. В Сессии 7 (Админка) появится полноценная сводка с фильтрами по периоду и группировкой; на главной — только моментальная метрика
- **`teamBackendClient` сейчас экспортирует только `fetchKeysStatus[Safe]()`.** Это намеренный минимализм для каркасной сессии — чтобы добавить запуск задач/правки в Сессии 6, нужно дополнить файл новыми экспортами поверх `backendFetch()`, без переписывания инфраструктуры
- **Build локально требует `frontend/.env.local`.** Если переменных `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` нет, `next build` падает на этапе collect page data. На Vercel переменные есть — деплой проходит. Это ограничение существовало и до Сессии 28 (страница `/blog/references` ведёт себя так же), просто теперь страниц с этим импортом стало больше

Ручные шаги для Влада (после пуша):
1. **Никаких миграций БД, никаких изменений в Storage, никаких новых env-переменных на Vercel.** Деплой — чисто фронтовой
2. После пуша Vercel автоматически передеплоит фронт (1–2 минуты от пуша)

## Известные ограничения и заметки
- Instagram API закрыт — используем Apify (платный, pay-as-you-go)
- Массовая обработка: 1 видео ≈ 60–120 сек (Apify+Whisper+AI). При workerConcurrency=2 пачка из 1000 займёт ~8 часов. Hard-лимит UI на пачку — 2000 ссылок (`MAX_BATCH_SIZE` в `frontend/src/lib/reelsUrlParser.ts`)
- Коэффициент вирусности: формулу уточним по ходу (соотношение лайков/комментов к просмотрам, нормированное по среднему для аккаунта)
- Транскрипция: только речь. Текст на экране (субтитры, наложенный текст) не распознаём — усложнение без большой ценности на данном этапе
- **Whisper API лимит 25 МБ.** Видео Reels часто превышают, поэтому в `transcriptionService` перед отправкой ffmpeg вытаскивает аудио в mp3 64 kbps моно — размер падает в 20–50 раз. Бинарь ffmpeg ставится пакетом `ffmpeg-static`, никаких системных установок не нужно
- **Шеры (количество отправок Reels)**: дефолтный актор `apify/instagram-scraper` поле не отдаёт. Альтернатива — `apify/instagram-reel-scraper` (специализированный, возвращает `sharesCount`); чтобы переключиться, поставить `APIFY_ACTOR_ID=apify/instagram-reel-scraper` в `backend/.env` и перезапустить бэкенд. В дефолтной конфигурации шеры остаются `null`, формула вирусности корректно работает без них
- **Превью с Instagram CDN**: ссылки временные (~24 часа) и требуют отсутствия Referer (на фронте уже стоит `referrerPolicy="no-referrer"`). Решено в Сессии 20 — после парсинга бэкенд перезаливает превью в публичную папку на Google Drive и хранит постоянный URL. Если переменные `GOOGLE_DRIVE_*` в env не заданы — система продолжает работать со ссылками Instagram CDN как раньше.
- Бэкенд локально — запускать через `npm start` (без `--watch`): `node --watch` может прервать обработку Apify на полпути и оставить строку в статусе `processing` навсегда. `npm run dev` оставлен для будущей доработки кода без активных задач Apify

## Правила архитектуры кода — ОБЯЗАТЕЛЬНО

### Модульность
- **НИКОГДА** не складывать весь код в один файл. Каждый компонент, каждый сервис, каждая утилита — отдельный файл.
- Один React-компонент = один файл. Максимум 150–200 строк на файл. Если больше — разбить на подкомпоненты.
- Бизнес-логика отделена от UI: компоненты не содержат прямых вызовов к API или базе. Вся логика — в `lib/` или `services/`.
- Общие UI-элементы (кнопки, модалки, инпуты) — в `components/ui/`, переиспользуются везде.

### Именование
- Файлы компонентов: PascalCase (`VideoTable.tsx`, `AddVideoModal.tsx`)
- Файлы утилит и сервисов: camelCase (`apifyService.ts`, `supabaseClient.ts`)
- Папки: kebab-case (`blog-analyst/`)

### Типизация
- TypeScript обязателен. Интерфейсы для всех структур данных (Video, Tag, Category и т.д.) — в отдельном файле `types.ts`.
